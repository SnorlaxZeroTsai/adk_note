## Part 1：基礎概念（給完全零基礎的人）

### 認證 vs 授權

| | 認證（Authentication） | 授權（Authorization） |
|---|---|---|
| 問題 | 「你是誰？」 | 「你能做什麼？」 |
| 類比 | 進大樓刷門禁卡 | 刷卡後只能進自己樓層 |
| 失敗結果 | 401 Unauthorized | 403 Forbidden |
| 發生順序 | 先 | 後 |

### 什麼是 API Key？

最簡單的認證方式——一串隨機字串，像密碼一樣：

```
sk-abc123xyz789def456...
```

- **你有這串字串** = 你就是這個 key 的主人
- 跟密碼的差異：密碼是人記的（短、有規律），API Key 是機器生的（長、隨機）
- 放在 HTTP header 裡送出：

```http
POST /v1/chat/completions HTTP/1.1
Authorization: Bearer sk-abc123xyz789def456...
Content-Type: application/json
```

### 什麼是 JWT？

JWT（JSON Web Token）是一個**自帶證明的身份證**：

```
eyJhbGciOiJSUzI1NiJ9.eyJzdWIiOiJ1c2VyLTEyMyIsInRlYW1faWQiOiJ0ZWFtLWFiYyJ9.SIGNATURE
```

由三段用 `.` 分隔：

```
┌──────────────┐   ┌──────────────────────────┐   ┌──────────────┐
│   Header     │ . │       Payload            │ . │  Signature   │
│ {"alg":"RS256"}│   │ {"sub":"user-123",      │   │ (加密簽名)    │
│              │   │  "team_id":"team-abc",   │   │              │
│              │   │  "exp":1735689600}       │   │              │
└──────────────┘   └──────────────────────────┘   └──────────────┘
```

- **Header**：用什麼演算法簽名
- **Payload**：包含身份資訊（誰、屬於哪個 team、什麼時候過期）
- **Signature**：用私鑰簽名，防偽造

**類比**：
- API Key = 鑰匙（拿到就能進門，不知道你是誰）
- JWT = 附照片的證件（自帶身份資訊，可以驗真偽）

### 什麼是 OAuth2？

當你看到「用 Google 登入」，那就是 OAuth2：

```
1. 使用者點「用 Google 登入」
2. 跳轉到 Google 頁面輸入密碼
3. Google 驗證後給你一個 token
4. 你拿 token 回到原本的系統
5. 系統拿 token 去問 Google：「這人是誰？」
6. Google 回答：「是 john@gmail.com」
```

- 你的系統**永遠不碰使用者的密碼**
- Google 是「身份提供者」（Identity Provider）

### 什麼是 RBAC？

RBAC（Role-Based Access Control）= 按角色分配權限：

```
角色: ADMIN
  → 可以：建立 team、刪除 key、看所有資料

角色: TEAM
  → 可以：呼叫 LLM、看自己 team 的花費
  → 不可以：看別人的 team、建立新 team

角色: USER
  → 可以：呼叫 LLM
  → 不可以：管理任何東西
```

不用為每個人設定權限，只要指定角色就好。

### 什麼是 Hash？

```python
hash("sk-abc123xyz789") → "a7f3b2c1d4e5f6..."  # 不可逆的「指紋」
```

- 相同輸入 → 永遠相同輸出
- 不可能從 hash 反推出原始值
- LiteLLM **只存 hash**，不存原始 key → 資料庫被偷也沒用

### 憑證提取優先順序

LiteLLM 從多個地方找 token（按優先順序）：

```python
# 1. 自定義 header
request.headers.get("X-LiteLLM-Key")

# 2. 標準 Bearer token
request.headers.get("Authorization")  # "Bearer sk-xxx"

# 3. Azure 格式
request.headers.get("api-key")

# 4. Anthropic 格式
request.headers.get("x-api-key")
```

### 公開路由（不需要認證）

這些 endpoint 不需要 API key：

```
/health          → 健康檢查
/health/liveliness → 存活檢查
/health/readiness  → 就緒檢查
```

## Part 3：API Key（Virtual Key）系統

### 什麼是 Virtual Key？

LiteLLM 的 key 不是直接連到 OpenAI/Anthropic 的 key，而是 Proxy 自己發的「虛擬 key」：

```
使用者 → sk-litellm-abc (虛擬 key) → LiteLLM Proxy → sk-openai-xyz (真正的 key) → OpenAI
```

好處：
- 真正的 Provider key 只存在 Proxy 裡，使用者看不到
- 一個 Provider key 可以給多個虛擬 key 共用
- 虛擬 key 可以限制模型、預算、速率

### Key 的生命週期

```
建立                          使用                          失效
 │                            │                            │
 ▼                            ▼                            ▼
POST /key/generate          每次請求                    過期 or 預算用完
 │                            │                            │
 ├─ 生成隨機字串               ├─ hash(key) → 查 DB         ├─ expires 到了
 │  "sk-" + 32 chars          ├─ 檢查 team/budget/model    ├─ spend >= max_budget
 ├─ hash → 存 DB              ├─ 放行 → 呼叫 LLM           └─ 手動 blocked=true
 └─ 回傳明文給使用者           └─ 記錄花費
    （唯一一次看到明文）
```

### 資料庫結構

```sql
-- LiteLLM_VerificationToken（存 hash 後的 key）
CREATE TABLE LiteLLM_VerificationToken (
    token       TEXT PRIMARY KEY,      -- SHA-256(sk-xxx)
    key_name    TEXT,                   -- "Alice 的開發用 key"
    team_id     TEXT,                   -- 屬於哪個 team
    user_id     TEXT,                   -- 屬於哪個 user
    models      JSON,                   -- ["gpt-4", "gpt-3.5-turbo"]
    max_budget  FLOAT,                  -- 100.0（美元）
    spend       FLOAT DEFAULT 0,       -- 目前已花
    rpm_limit   INT,                    -- 1000 requests/min
    tpm_limit   INT,                    -- 100000 tokens/min
    expires     TIMESTAMP,             -- 何時過期
    blocked     BOOLEAN DEFAULT FALSE, -- 是否停用
    metadata    JSON                   -- 自定義資料
);
```

### 為什麼用 Hash 不存明文？

```
場景：資料庫被駭客偷了

存明文：駭客直接拿所有 key 去用 → 災難
存 hash：駭客拿到一堆 hash → 沒用（不可逆）

安全比較也很重要：
❌ if provided_key == stored_key:     # 時間側通道攻擊
✅ secrets.compare_digest(a, b):      # 固定時間比較
```

### 建立 Key 的範例

```bash
# 建立一個限制模型和預算的 key
curl -X POST http://localhost:4000/key/generate \
  -H "Authorization: Bearer sk-master-key" \
  -H "Content-Type: application/json" \
  -d '{
    "team_id": "team-backend",
    "key_name": "Alice 開發用",
    "models": ["gpt-4o", "gpt-4o-mini"],
    "max_budget": 50.0,
    "rpm_limit": 500,
    "tpm_limit": 50000,
    "duration": "30d"
  }'

# 回應
{
  "key": "sk-litellm-abc123xyz...",     # 明文（只出現這一次！）
  "key_name": "Alice 開發用",
  "expires": "2024-07-22T00:00:00Z",
  "max_budget": 50.0
}
```

## Part 4：JWT 認證

### 什麼時候用 JWT？

- **API Key**：適合 server-to-server 通訊（後端呼叫後端）
- **JWT**：適合有「使用者登入」的場景（使用者透過身份提供者登入後拿到 JWT）

```
使用者 → 登入公司 SSO（Okta/Auth0/Google）→ 拿到 JWT
     → 用 JWT 呼叫 LiteLLM Proxy
     → Proxy 用公鑰驗證 JWT 是否合法
     → 不需要查資料庫！（JWT 自帶身份資訊）
```

### JWT 驗證流程

```python
# Step 1：解碼 header（不驗簽名，只是看用什麼演算法和 key ID）
header = jwt.get_unverified_header(token)
# {"alg": "RS256", "kid": "key-id-123"}

# Step 2：去身份提供者拿公鑰
# JWKS URL 例如: https://auth.company.com/.well-known/jwks.json
jwks = httpx.get("https://auth.company.com/.well-known/jwks.json")
public_key = find_key_by_kid(jwks, header["kid"])

# Step 3：驗證簽名 + 過期時間 + audience + issuer
payload = jwt.decode(
    token,
    key=public_key,
    algorithms=["RS256"],
    audience="litellm-proxy",        # 這個 JWT 是發給我的嗎？
    issuer="https://auth.company.com", # 是可信的發行者嗎？
)
# payload = {"sub": "user-123", "team_id": "team-abc", "exp": 1735689600}

# Step 4：從 payload 取出身份
user_id = payload["sub"]
team_id = payload["team_id"]
```

### JWKS 是什麼？

JWKS（JSON Web Key Set）= 公鑰的公開目錄：

```json
// https://auth.company.com/.well-known/jwks.json
{
  "keys": [
    {
      "kty": "RSA",
      "kid": "key-id-123",        // Key ID（對應 JWT header 的 kid）
      "use": "sig",               // 用途：簽名
      "n": "0vx7agoebGc...",     // RSA 公鑰的 n 值
      "e": "AQAB"                // RSA 公鑰的 e 值
    }
  ]
}
```

**為什麼用公鑰/私鑰？**

```
身份提供者（Okta）  →  私鑰簽名 JWT  →  只有 Okta 能簽
LiteLLM Proxy      →  公鑰驗證 JWT  →  任何人都能驗

= 沒有人能偽造 JWT（因為沒有私鑰）
= LiteLLM 不需要跟 Okta 溝通就能驗證（離線驗證）
```

### LiteLLM 的 JWT 設定

```yaml
# proxy config.yaml
litellm_settings:
  enable_jwt_auth: true

general_settings:
  litellm_jwtauth:
    # 身份提供者的公鑰 URL
    public_key_jwks_url: "https://auth.company.com/.well-known/jwks.json"

    # JWT 中哪個欄位對應什麼身份
    user_id_jwt_field: "sub"          # "sub" 欄位 = user_id
    team_id_jwt_field: "team_id"      # "team_id" 欄位 = team_id
    org_id_jwt_field: "org"           # "org" 欄位 = org_id
    user_email_jwt_field: "email"

    # Admin 判斷（有這個 scope 就是 admin）
    admin_jwt_scope: "litellm:admin"

    # 期望的 audience（防止 JWT 被用在錯的服務）
    audience: "litellm-proxy"
```

### JWT vs API Key 比較

| 面向 | API Key | JWT |
|------|---------|-----|
| 儲存位置 | 資料庫 | 不需要存（自帶資訊） |
| 驗證方式 | hash → 查 DB | 公鑰驗簽名 |
| 吊銷方式 | DB 設 blocked=true | 無法即時吊銷（等過期） |
| 效能 | 需要 DB 查詢 | 純 CPU 運算（快） |
| 適用場景 | 長期 server-to-server | 短期使用者 session |
| 安全性 | 洩漏了就要換 | 自動過期（通常 1 小時） |

### LiteLLM 的角色層次

```
┌────────────────────────────────────────────────┐
```

### 角色判定邏輯

```python
def determine_role(token_or_jwt):
    # JWT 的情況
    if has_admin_scope(jwt):
        return PROXY_ADMIN

    if has_team_id(jwt):
        return TEAM

    if has_user_id(jwt):
        return INTERNAL_USER

    # API Key 的情況：查 DB 中 user 的 user_role 欄位
    user = get_user(key.user_id)
    return user.user_role  # "proxy_admin" | "internal_user" | ...
```

## Part 6：多租戶隔離（Team/Org Scoping）

### 架構

```
Organization "Acme Corp" (org_id: org-acme)
│
├── Team "Backend" (team_id: team-backend)
│   ├── Alice (user_id: alice)
│   │   ├── Key: sk-alice-1 (max_budget: $50)
│   │   └── Key: sk-alice-2 (max_budget: $20, models: [gpt-4o-mini])
│   └── Bob (user_id: bob)
│       └── Key: sk-bob-1 (max_budget: $100)
│
└── Team "ML" (team_id: team-ml)
    ├── Charlie (user_id: charlie)
    │   └── Key: sk-charlie-1 (max_budget: $500)
    └── David (user_id: david)
        └── Key: sk-david-1 (max_budget: $200)
```

### 隔離原則

Alice 的 key `sk-alice-1` 只能：
- 呼叫 team-backend 允許的模型
- 花費不超過自己的 $50 限額
- 花費不超過 team-backend 的團隊總預算
- 看到 team-backend 的花費資訊

Alice **不能**：
- 看到 team-ml 的任何資料
- 用 Charlie 的模型額度
- 呼叫 team-ml 才有權限的模型

### 預算階層

每一層都是獨立的限制，**最嚴格的那個生效**：

```
檢查順序（任一層超過就擋）：

1. Key 預算：     sk-alice-1 → max_budget: $50, spend: $48
2. 團隊成員預算： Alice in team-backend → max_budget_in_team: $100
3. Team 預算：    team-backend → max_budget: $500, spend: $480
4. Org 預算：     org-acme → max_budget: $5000

Alice 的下一個請求會被擋嗎？
→ Key 預算：$48 < $50 ✅
→ Team 預算：$480 < $500 ✅
→ 放行！

但如果 Team spend 已經 $500：
→ Team 預算：$500 >= $500 ❌
→ 擋住！（即使 Alice 自己還有額度）
```

### 資料庫表

```sql
-- 團隊成員資格（含獨立預算）
CREATE TABLE LiteLLM_TeamMembership (
    user_id     TEXT,
    team_id     TEXT,
    role        TEXT,          -- "admin" or "member"
    max_budget  FLOAT,         -- 此人在此 team 的預算
    spend       FLOAT,         -- 此人在此 team 已花費
    tpm_limit   INT,
    rpm_limit   INT,
    PRIMARY KEY (user_id, team_id)
);
```

## Part 7：速率限制（Rate Limiting）

### 兩種限制

```
RPM (Requests Per Minute)
─────────────────────────
「一分鐘內最多發幾個請求」
例：rpm_limit=100 → 每分鐘最多 100 個 API 呼叫
超過 → 429 Too Many Requests

TPM (Tokens Per Minute)
─────────────────────────
「一分鐘內最多用多少 token」
例：tpm_limit=100000 → 每分鐘最多 10 萬 token
超過 → 429 Too Many Requests
```

### 為什麼需要兩種？

```
場景 A：短請求但量大
  100 個請求 × 每個 100 token = 10000 token/min
  → RPM 先爆（100 次），TPM 還很多
  → RPM 是瓶頸

場景 B：少量但超長請求
  5 個請求 × 每個 50000 token = 250000 token/min
  → TPM 先爆（25 萬），RPM 才 5
  → TPM 是瓶頸
```

### 計數方式

```python
# 每分鐘一個 window，自動過期
cache_key = f"{key_hash}:rpm:{hour}-{minute}"  # TTL = 60s

# Pre-Call：請求進來時
current_rpm = cache.get(key)
if current_rpm >= rpm_limit:
    raise HTTPException(429, headers={"Retry-After": "60"})
cache.increment(key)

# Post-Call：請求完成時
tokens_used = response.usage.total_tokens
cache.increment(f"{key_hash}:tpm:{hour}-{minute}", tokens_used)
```

### 多實例同步

多個 Proxy 實例共享限制（透過 Redis）：

```
Instance A: 本地計數 rpm=30
Instance B: 本地計數 rpm=40
                    ↓
              Redis 匯總：rpm=70
                    ↓
            limit=100 → 還有空間
```

## Part 8：IP 白名單

### 設定

```yaml
general_settings:
  allowed_ips: ["192.168.1.0/24", "10.0.0.5"]
  use_x_forwarded_for: true  # 如果在 load balancer 後面
```

### 檢查邏輯

```python
def check_ip(request):
    # 在 reverse proxy（Nginx/ALB）後面時，真實 IP 在 header 裡
    if use_x_forwarded_for:
        client_ip = request.headers["x-forwarded-for"].split(",")[0]
    else:
        client_ip = request.client.host

    if client_ip not in allowed_ips:
        raise HTTPException(403, f"IP {client_ip} not allowed")
```

### 常見場景

| 場景 | 設定 |
|------|------|
| 只允許公司 VPC | `["10.0.0.0/8"]` |
| 只允許辦公室 IP | `["203.0.113.50"]` |
| 允許 Cloudflare | `["173.245.48.0/20", "103.21.244.0/22", ...]` |

## Part 9：OAuth2 Token Introspection

### 跟 JWT 的差異

```
JWT：token 自帶資訊 → 本地驗證 → 無法即時吊銷
OAuth2：token 只是一串 ID → 要問發行者「這 token 還有效嗎？」
```

### 流程

```python
# 1. 使用者送 token
headers = {"Authorization": "Bearer oacc_abc123xyz"}

# 2. Proxy 去問 OAuth2 provider
response = httpx.post(
    "https://oauth-provider.com/oauth/introspect",
    data={"token": "oacc_abc123xyz"},
    auth=(client_id, client_secret),
)

# 3. Provider 回答
{
    "active": true,        # token 是否有效
    "sub": "user-789",     # 使用者 ID
    "scope": "read write", # 權限範圍
    "exp": 1735689600      # 過期時間
}

# 4. 如果 active=false → 拒絕
# 5. 如果 active=true → 提取身份繼續
```

### 什麼時候用 OAuth2？

- 需要**即時吊銷** token（安全事件時立刻生效）
- 已有 OAuth2 基礎設施（公司用 Okta/Auth0 管理員工）
- Token 需要**動態權限**（provider 可以隨時更改回應）

## Part 10：快取系統（為什麼 auth 很快？）

### 問題

每個請求都做 auth = 每個請求都查資料庫 5+ 次：
1. 查 key → 1 次
2. 查 team → 1 次
3. 查 user → 1 次
4. 查 org → 1 次
5. 查 membership → 1 次

1000 QPS × 5 次 = 每秒 5000 次 DB 查詢 → 太慢了

### 解法：分層快取

```
請求進來
  → L1: 進程記憶體 (< 0.1ms) → 命中！直接返回
  → L2: Redis (~ 1ms) → 命中！回填 L1，返回
  → L3: PostgreSQL (~ 10-50ms) → 回填 L1 + L2，返回
```

### Cache Key 格式

```
"team_id:team-backend"     → Team 物件
"user_id:alice"            → User 物件
"sk-hash-abc123..."        → Key 權限物件
"end_user_id:customer-1"   → End User 物件
```

### 快取失效

```python
# 更新 team 時
await db.update_team(team_id="team-backend", max_budget=1000)
await cache.delete("team_id:team-backend")  # 清除快取
# 下次查詢 → cache miss → 從 DB 重新載入
```

## Part 11：安全防護

### 禁止的請求參數

有些參數如果使用者能設，就能攻擊系統：

```python
BANNED_PARAMS = [
    "api_base",           # 使用者不能指定要打哪個 URL（防止 SSRF）
    "azure_ad_token",     # 不能注入 Azure 認證
    "langfuse_host",      # 不能把日誌導到自己的伺服器（洩漏資料）
    "aws_sts_endpoint",   # 不能觸發 AWS API 呼叫
]

# 如果請求 body 包含這些 → 直接拒絕
```

### API Key 遮罩

```python
# 日誌中永遠不出現完整 key
def abbreviate_key(key: str) -> str:
    return key[:5] + "****" + key[-3:]
    # "sk-abc123xyz" → "sk-ab****xyz"
```

### 時間側通道防護

```python
# ❌ 普通比較：字串越長、匹配越多就越慢 → 洩漏資訊
if user_key == stored_key:

# ✅ 固定時間比較：不管匹配多少，花的時間一樣
import secrets
secrets.compare_digest(user_key, stored_key)
```

## Part 12：完整範例（把所有概念串起來）

```bash
# 使用者發送請求
curl -X POST http://proxy.company.com/v1/chat/completions \
  -H "Authorization: Bearer sk-alice-abc123" \
  -H "Content-Type: application/json" \
  -d '{"model": "gpt-4o", "messages": [{"role": "user", "content": "Hello"}]}'
```

**Proxy 內部發生什麼：**

```python
# 1. 提取 token
token = "sk-alice-abc123"

# 2. 不是 JWT（沒有兩個 "."） → 走 API Key 路徑

# 3. Hash
token_hash = sha256("sk-alice-abc123")  # → "a7f3b2..."

# 4. 查快取
key_obj = cache.get("a7f3b2...")
# 命中！不需要查 DB
# key_obj = {
#   team_id: "team-backend",
#   user_id: "alice",
#   models: ["gpt-4o", "gpt-4o-mini"],
#   max_budget: 50.0,
#   spend: 35.0,
#   rpm_limit: 500,
#   tpm_limit: 50000,
# }

# 5. 檢查 model 是否允許
"gpt-4o" in ["gpt-4o", "gpt-4o-mini"]  # ✅

# 6. 檢查 key 預算
35.0 < 50.0  # ✅ 還有 $15

# 7. 查 team 預算
team = cache.get("team_id:team-backend")
team.spend < team.max_budget  # $400 < $500 ✅

# 8. 檢查 RPM
current_rpm = cache.get("a7f3b2...:rpm:14-23")  # 這分鐘第 42 個
42 < 500  # ✅

# 9. 全部通過！呼叫 LLM
response = await openai.chat.completions.create(model="gpt-4o", ...)

# 10. 記錄花費（非同步，不阻塞回應）
asyncio.create_task(record_spend(
    key="a7f3b2...",
    team="team-backend",
    user="alice",
    cost=0.003,
    tokens=150,
))

# 11. 回傳給使用者
return response
```

## Part 13：設計亮點總結

| 設計 | 為什麼好 |
|------|---------|
| Hash 儲存 key | DB 被偷也沒用 |
| 分層快取 | auth 延遲 < 1ms |
| 預算階層（Key→User→Team→Org） | 精細控制，最嚴格的生效 |
| JWT 離線驗證 | 不需要查 DB，高吞吐 |
| RBAC 角色 | 不用為每人設權限 |
| Virtual Key | 真正的 provider key 不暴露 |
| Team 隔離 | 多租戶完全隔離 |
| IP 白名單 | 網路層防護 |
| 固定時間比較 | 防時間側通道攻擊 |
| 禁止危險參數 | 防 SSRF/資料洩漏 |
