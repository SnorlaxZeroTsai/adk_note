## 策略總覽

| 策略 | 演算法 | 延遲 | 適用場景 |
|------|--------|------|---------|
| `simple-shuffle` | 加權隨機 | <0.1ms | 通用，零配置 |
| `least-busy` | 追蹤 in-flight 請求數 | <0.1ms | 避免單點過載 |
| `lowest-latency` | 移動平均 per-token 延遲 | <1ms | 最快回應 |
| `lowest-cost` | 靜態 $/token 比較 | <0.1ms | 省錢 |
| `lowest-tpm-rpm-v2` | 分鐘級 token/request 計數 | <1ms | 尊重 Provider 限流 |
| `tag-based` | 標籤匹配 | <0.1ms | 多租戶/環境隔離 |
| `complexity-router` | 7 維度加權評分 | <1ms | 簡單問題用便宜模型 |
| `quality-router` | 複雜度 + 關鍵字覆寫 | <1ms | 按意圖分流 |
| `adaptive-router` | Thompson Sampling (Bandit) | ~5ms | 自動學習最佳模型 |

## 基礎架構：BaseRoutingStrategy

所有進階策略都繼承自 `BaseRoutingStrategy`，它提供：

```python
class BaseRoutingStrategy:
    """雙層 cache + 批次 Redis 寫入"""

    def _increment_value_in_current_window(self, key, value):
        """本地 cache 即時更新 + 排隊寫 Redis"""
        self.in_memory_cache.increment(key, value)
        self._redis_queue.append((key, value))

    async def _push_in_memory_increments_to_redis(self):
        """批次壓縮同 key 的多次 increment → 一次 Redis pipeline"""
        compressed = compress_operations(self._redis_queue)
        await self.redis_cache.pipeline_increment(compressed)

    async def periodic_sync_in_memory_spend_with_redis(self):
        """背景任務：每 10 秒同步 Redis → 本地"""
        while True:
            await asyncio.sleep(10)
            await self._sync_in_memory_spend_with_redis()
```

**設計要點**：
- 讀取：本地 cache（O(1)，零延遲）
- 寫入：先寫本地，再批次 flush 到 Redis
- 同步：背景任務定期拉 Redis 更新（跨實例一致性）
- 壓縮：同一 key 的多次 +1 合併為一次 +N

```python
def get_available_deployment(self, healthy_deployments):
```

## 2. Least Busy（最少忙碌）

追蹤每個部署的 in-flight 請求數，選最少的。

```python
# 生命週期
pre_call:  request_count[deployment_id] += 1
success:   request_count[deployment_id] -= 1
failure:   request_count[deployment_id] -= 1

# 路由
def route(self, deployments):
    return min(deployments, key=lambda d: self.request_count[d.id])
```

**適用場景**：部署效能相近，但某些請求特別慢（長 completion），避免所有慢請求都堆到同一台。

## 3. Lowest Latency（最低延遲）

追蹤每個部署最近 N 次的 per-token 延遲：

```python
# 資料結構：Ring Buffer（最多 10 筆）
{
    "deployment_1": {
        "latency": [0.32, 0.28, 0.35, 0.30, ...],        # 秒/token
        "time_to_first_token": [0.05, 0.04, 0.06, ...],   # 串流首 token
    }
}

# 延遲計算
non_streaming: latency = response_time / completion_tokens
streaming:     latency = time_to_first_token
timeout:       latency = 1000.0  # 懲罰值

# 路由
avg_latency = mean(ring_buffer)
lowest = min(all_avg_latencies)
valid = [d for d in deployments if d.avg_latency <= lowest * (1 + buffer)]
return random.choice(valid)  # buffer 範圍內隨機選
```

**設定**：

```yaml
routing_strategy: "lowest_latency"
routing_args:
  lowest_latency_buffer: 0.1    # 10% 容忍範圍
  max_latency_list_size: 10     # Ring buffer 大小
```

**為什麼用 Ring Buffer 不用全部歷史？**
- 延遲會隨時間變化（網路波動、伺服器負載）
- 只看最近 10 次 = 快速適應變化
- 全部歷史 = 舊資料拖慢反應速度

## 4. Lowest Cost（最低成本）

直接比較各部署的 $/token：

```python
def get_cost(deployment):
    input_cost = deployment.get("input_cost_per_token") or model_cost_map.get(model, 5.0)
    output_cost = deployment.get("output_cost_per_token") or model_cost_map.get(model, 5.0)
    return input_cost + output_cost

def route(self, deployments):
    return min(deployments, key=get_cost)
```

**適用場景**：相同模型有不同定價的部署（如 Azure vs OpenAI 直連，不同 region 價格不同）。

## 5. Lowest TPM/RPM v2（速率限制感知）

尊重每個部署的 TPM/RPM 上限，在多個 Gateway 實例間共享計數。

```python
# Cache Key（每分鐘一個 window）
key = f"{model_id}:{model_name}:rpm:{hour}-{minute}"

# Pre-Call 檢查
current_rpm = await cache.get(key)
if current_rpm >= deployment.rpm_limit:
    raise RateLimitError(f"rpm_limit={limit}, current={current_rpm}")

# Post-Call 更新
await cache.increment(f"...tpm:...", response.usage.total_tokens)
await cache.increment(f"...rpm:...", 1)
# TTL = 60 秒（每分鐘自動重置）
```

**多實例同步**：

```
Instance A: local_rpm=5, Redis 查到其他實例已用 45 → 總計 50 → 接近 limit
Instance B: local_rpm=3, 同樣查 Redis → 一致的全域視圖
```

**設定**：

```yaml
model_list:
  - model_name: openai-gpt4
    litellm_params:
      model: openai/gpt-4o
      tpm: 90000     # Provider 給的上限
      rpm: 3500
```

## 7. Complexity Router（複雜度路由）

**核心想法**：簡單問題（「什麼是 HTTP」）用便宜模型，複雜問題（「實作一個分散式鎖」）用強模型。

### 7 維度評分

| 維度 | 權重 | 偵測方式 |
|------|------|---------|
| tokenCount | 0.10 | 字數估算（÷4） |
| codePresence | 0.30 | 程式碼關鍵字（function, class, async...） |
| reasoningMarkers | 0.25 | 推理指示詞（"step by step", "analyze"...） |
| technicalTerms | 0.25 | 技術術語（architecture, distributed...） |
| simpleIndicators | 0.05 | 簡單指示詞（"what is", "define"...）**負權重** |
| multiStepPatterns | 0.03 | 多步驟模式（"first...then..."） |
| questionComplexity | 0.02 | 問號數量 |

### 計分方式

```python
def classify(self, messages):
    user_text = extract_last_user_message(messages)
    scores = {}

    # Token 數量
    tokens = len(user_text) // 4
    if tokens < 15:
        scores["tokenCount"] = -1.0    # 很短 = 簡單
    elif tokens > 400:
        scores["tokenCount"] = 1.0     # 很長 = 可能複雜
    else:
        scores["tokenCount"] = 0.0

    # 程式碼關鍵字
    code_hits = count_keyword_matches(user_text, CODE_KEYWORDS)
    scores["codePresence"] = 0.5 if code_hits == 1 else (1.0 if code_hits >= 2 else 0.0)

    # ... 其他維度類似 ...

    # 加權總分
    total = sum(scores[dim] * WEIGHTS[dim] for dim in scores)
    return total

# 分數 → 模型層級
SIMPLE:    score < 0.15   → gpt-4o-mini
MEDIUM:    0.15 ~ 0.35    → gpt-4o
COMPLEX:   0.35 ~ 0.60    → claude-sonnet-4
REASONING: score ≥ 0.60   → o1-preview
```

### 特殊規則：推理覆寫

如果偵測到 2+ 個推理指示詞，無論總分多低都直接路由到 REASONING 層級：

```python
if reasoning_marker_count >= 2:
    return Tier.REASONING  # 使用者明確要求深度思考
```

### 設定範例

```yaml
model_list:
  - model_name: smart-router
    litellm_params:
```

- 分類延遲：< 1ms（純 regex + 計數，無外部 API）

- 無依賴：離線可用

## 8. Quality Router（品質路由）

複雜度路由的擴展版，加上**關鍵字覆寫**：

```python
def route(self, messages, deployments):
    user_text = extract_last_user_message(messages)

    # 1. 先檢查有沒有關鍵字匹配
    for deployment in deployments:
        keywords = deployment.model_info.get("keywords", [])
        if any(kw.lower() in user_text.lower() for kw in keywords):
            return deployment  # 直接命中！

    # 2. 沒有關鍵字匹配 → 走複雜度分類
    tier = self.complexity_router.classify(messages)
    return self.get_deployment_for_tier(tier, deployments)
```

**設定**：

```yaml
model_list:
  - model_name: python-expert
    litellm_params:
      model: openai/gpt-4o
    model_info:
      litellm_routing_preferences:
        quality_tier: 3
        keywords: [python, code, function, def, import]

  - model_name: writing-expert
    litellm_params:
      model: anthropic/claude-sonnet-4
    model_info:
      litellm_routing_preferences:
        quality_tier: 3
        keywords: [essay, write, creative, story]
```

使用者問 Python 問題 → 直接走 python-expert，不需要分類。

## 9. Adaptive Router（自適應路由）— 最複雜的策略

### 核心演算法：Thompson Sampling（多臂吃角子老虎機）

**問題**：有 N 個模型，每個模型的「品質」未知且可能隨時間變化。怎麼在**探索（試新的）**和**利用（用已知好的）**之間平衡？

**答案**：為每個模型維護一個 Beta 分佈，用 Thompson Sampling 選模型。

### 資料結構

每個 `(request_type, model)` 維護一組 Beta 分佈參數：

```python
@dataclass(frozen=True)
class BanditCell:
    alpha: float   # 「成功」偽計數
    beta: float    # 「失敗」偽計數

    @property
    def mean(self) -> float:
        """期望品質 = alpha / (alpha + beta)"""
        return self.alpha / (self.alpha + self.beta)
```

### Thompson Sampling 選模型

```python
def pick_best(cells: Dict[str, BanditCell], model_costs: Dict[str, float]):
    scores = {}
    for model, cell in cells.items():
        # 1. 從 Beta(alpha, beta) 抽一個樣本
        quality_sample = random.betavariate(cell.alpha, cell.beta)

        # 2. 成本正規化（最貴=0, 最便宜=1）
        cost_score = normalize_cost(model_costs[model])

        # 3. 加權分數
        scores[model] = 0.7 * quality_sample + 0.3 * cost_score

    return max(scores, key=scores.get)
```

**為什麼用 Thompson Sampling 不用 UCB？**

| 演算法 | 探索方式 | 問題 |
|--------|---------|------|
| ε-greedy | 固定機率隨機探索 | 探索不夠聰明（已知差的也會選到） |
| UCB | 信心上界 → 選不確定的 | 確定性選擇，無隨機性 → 陷入局部最優 |
| Thompson Sampling | 從後驗分佈抽樣 | ✅ 自然平衡探索/利用，已證明漸近最優 |

### 冷啟動先驗

新模型沒有歷史資料，用先驗知識初始化：

```python
def initial_cell(model_preferences):
    # 根據模型宣告的 quality_tier 設定初始 mean
    base = {1: 0.3, 2: 0.5, 3: 0.7}[quality_tier]

    # 如果模型宣告擅長此類型請求，加分
    bonus = 0.3 if request_type in model.strengths else 0.0

    mean = min(0.95, base + bonus)

    # COLD_START_MASS = 10（約 10 次真實觀測就能移動後驗）
    alpha = mean * 10
    beta = (1 - mean) * 10
    return BanditCell(alpha=alpha, beta=beta)
```

### 請求類型分類

將使用者 prompt 分成 7 種類型（用 regex 偵測）：

```
code_generation        → "write a function", "implement", code blocks
writing                → "write an essay", "summarize"
analytical_reasoning   → "analyze", "compare", "evaluate"
factual_lookup         → "what is", "when did", "who was"
math_and_computation   → "calculate", numbers, equations
creativity_and_brainstorm → "brainstorm", "ideas for"
general                → 其他
```

每個 `(request_type, model)` 有獨立的 Bandit Cell，讓系統學到「GPT-4o 擅長 code，Claude 擅長 writing」。

### 信號偵測（更新 Bandit）

每次 completion 結束後，分析回應品質：

```python
def compute_signals(session_state, response):
    signals = SignalDelta()

    # 正面信號（alpha += 1）
    if user_seems_satisfied(response):        # "thanks", "perfect", "yes"
        signals.satisfaction += 1

    # 負面信號（beta += 1）
    if response_misaligned(request, response):  # 回應與問題不匹配
        signals.misalignment += 1
    if response_too_similar_to_last(response):  # Jaccard > 0.5
        signals.stagnation += 1
    if user_disengages():                       # 使用者放棄對話
        signals.disengagement += 1
    if api_error():                             # 504, timeout
        signals.failure += 1
    if tool_call_loop_detected():               # 同一工具呼叫 > N 次
        signals.loop += 0.5

    return signals

# 更新
d_alpha = signals.satisfaction
d_beta = signals.misalignment + signals.stagnation + signals.failure + 0.5 * signals.loop

new_cell = BanditCell(
    alpha=min(cell.alpha + d_alpha, SAMPLE_CAP),
    beta=min(cell.beta + d_beta, SAMPLE_CAP),
)
# SAMPLE_CAP = 200（防止過度自信）
```

### Owner Cache（跨 turn 歸因）

多輪對話中，每一 turn 可能路由到不同模型。誰「負責」使用者的滿意/不滿意？

```
Turn 1: GPT-4o 回答 → 使用者說 "thanks" → 誰得分？
Turn 2: Claude 回答 → 使用者放棄 → 誰扣分？
```

**解法**：Owner Cache

```python
# 第一次選模型時，記錄 "owner"
owner_cache[session_id] = "gpt-4o"  # TTL 24h

# 後續 turn：只有 owner 模型的 bandit 會被更新
if current_model == owner_cache[session_id]:
    update_bandit(current_model, signals)
else:
    skip()  # 其他模型服務的 turn 不更新（避免錯誤歸因）
```

### 已知限制（v0）

| 限制 | 影響 |
|------|------|
| 信號純靠 regex | 無法深度理解使用者滿意度 |
| SAMPLE_CAP=200 後靜默丟棄 | 長期使用後停止學習 |
| Owner TTL 24h 無主動淘汰 | 記憶體可能緩慢增長 |
| 不考慮延遲 | 只看 quality + cost |
| Delta mapping 未驗證 | satisfaction=+1, failure=+1 的比例是猜的 |

## 選擇策略的決策樹

```
你的場景是什麼？
│
├── 「不知道用什麼」
│   └── simple-shuffle（零配置，夠用）
│
├── 「想省錢」
│   ├── 所有部署是同一模型 → lowest-cost
│   └── 不同模型可接受 → complexity-router（簡單問題用便宜模型）
│
├── 「想最快回應」
│   └── lowest-latency
│
├── 「Provider 有嚴格 rate limit」
│   └── lowest-tpm-rpm-v2
│
├── 「多租戶/多環境」
│   └── tag-based
│
├── 「想自動找到最佳模型」
│   └── adaptive-router（需要時間學習）
│
└── 「想根據 prompt 內容選模型」
    ├── 按複雜度 → complexity-router
    └── 按關鍵字/意圖 → quality-router
```

## 策略組合

實務中可以組合使用：

```
1. Tag-based 先按租戶分流
2. 同一租戶內用 complexity-router 按複雜度選模型
3. 同一複雜度層級的多個部署用 lowest-tpm-rpm-v2 分配
4. 全部疊上 cooldown cache 自動排除故障部署
```

```
Request
  → Tag Routing（租戶隔離）
    → Complexity Router（模型選擇）
      → TPM/RPM Router（負載分配）
        → Cooldown Filter（健康過濾）
          → 最終部署
```

| `litellm/router_strategy/adaptive_router/` | 自適應路由（Bandit） |
| `litellm/router_strategy/budget_limiter.py` | 預算限制器 |

## 快速複習

| 重點 | 一句話 |
|------|--------|
| BaseRoutingStrategy | 本地 cache 即時讀、批次 flush Redis、背景 10 秒同步 |
| Simple Shuffle | 加權隨機（weight/rpm/tpm），80% 場景夠用 |
| Lowest Latency | Ring buffer 10 筆移動平均，buffer 範圍內隨機 |
| Complexity Router | 7 維度評分（code/reasoning/technical 為主），<1ms 分類延遲 |
| Adaptive Router | Thompson Sampling + Beta 分佈，自動學習最佳模型 |
| Owner Cache | 多輪對話中只更新「首次服務者」的 Bandit，避免錯誤歸因 |
| 策略組合 | Tag → Complexity → TPM/RPM → Cooldown，層層篩選 |
