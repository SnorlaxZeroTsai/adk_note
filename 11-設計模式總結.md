# 11 - 設計模式總結

## 資深 Agent 工程師必須掌握的設計模式

這一章把前面學到的模式整理成可直接應用的知識。

## 模式 1：處理器管線（Processor Pipeline）

### 應用位置
Flow 中的 LLM 請求/回應處理

### 核心概念
```
Input → [P1] → [P2] → [P3] → ... → [Pn] → Output
```
每個處理器只負責一件事，可以獨立新增/移除/排序。

### 何時使用
- 請求/回應需要經過多個階段處理
- 不同場景需要不同的處理步驟組合
- 需要在不修改核心的情況下擴展功能

### ADK 中的應用
```python
# Flow 的請求處理器管線
processors = [
    basic_processor,      # 基礎設定
    auth_processor,       # 認證
    instruction_processor, # 指令注入
    content_processor,    # 內容準備
    cache_processor,      # 快取
]
# 每個處理器修改同一個 llm_request 物件
```

### 你自己的應用場景
```python
# 輸入前處理管線
def preprocess_pipeline(user_input):
    input = sanitize(user_input)      # 清理
    input = translate(input)          # 翻譯
    input = enrich_context(input)     # 添加 context
    return input
```

## 模式 2：Registry 模式（工廠 + 註冊表）

### 應用位置
Model 層的模型發現

### 核心概念
```python
# 註冊
registry.register("pattern", Constructor)

# 使用
instance = registry.get("name")  # 自動匹配 pattern，呼叫 Constructor
```

### 何時使用
- 多種實作，根據名稱/條件選擇
- 需要支援外部擴展（第三方註冊新實作）
- 惰性載入（用到才載入）

### ADK 中的應用
```python
# LLMRegistry
LLMRegistry.register(r"gemini-.*", GeminiLlm)
LLMRegistry.register(r"claude-.*", ClaudeLlm)

# 使用時
llm = LLMRegistry.get("gemini-2.5-flash")  # 自動用 GeminiLlm
```

### 你自己的應用場景
```python
# 工具 Registry
class ToolRegistry:
    _tools = {}
    
    @classmethod
    def register(cls, name, tool_class):
        cls._tools[name] = tool_class
    
    @classmethod
    def get(cls, name):
        return cls._tools[name]()

# 外部擴展：第三方套件可以在初始化時註冊工具
```

## 模式 3：回呼掛鉤（Callback Hooks）

### 應用位置
Agent 生命週期的各個階段

### 核心概念
```
[before_hook] → 核心邏輯 → [after_hook]
```
在不修改核心程式碼的情況下介入執行。

### 何時使用
- 需要在固定流程中注入自訂邏輯
- AOP（面向切面程式設計）需求
- 日誌、監控、權限檢查

### ADK 的多層回呼
```
before_agent → before_model → LLM → after_model → before_tool → 工具 → after_tool → after_agent
```

### 實戰技巧
```python
# 用 before_model_callback 做：
# - 輸入審核（阻止敏感請求）
# - 成本控制（超過預算就攔截）
# - A/B 測試（修改 model 參數）

# 用 after_tool_callback 做：
# - 結果快取
# - 結果驗證
# - 日誌記錄
```

## 模式 4：Event Sourcing

### 應用位置
Session State 管理、Workflow 恢復

### 核心概念
```
State = replay(all_events)  # 狀態是所有事件的累積結果
```

### 何時使用
- 需要完整的變更歷史
- 需要恢復到任意時間點
- 需要審計軌跡
- 需要中斷後恢復

### ADK 中的應用
```python
# State 不是直接存儲的，而是 Event 的 state_delta 累積
# Event 1: delta = {"user": "Alice"}
# Event 2: delta = {"score": 100}
# Event 3: delta = {"user": "Bob"}
# → State = {"user": "Bob", "score": 100}
```

## 模式 5：組合模式（Composition）

### 應用位置
多 Agent 系統架構

### 核心概念
```
複雜系統 = 組合多個簡單元件
```

### ADK 的三種組合方式

| 方式 | 機制 | 適用場景 |
|------|------|----------|
| Agent 轉移 | 動態切換 | 開放式對話 |
| Workflow 圖 | 靜態定義 | 確定性流程 |
| AgentTool | 工具呼叫 | 單次子任務 |

### 何時使用哪種

```
你的需求是什麼？
├── 需要 LLM 動態決定走哪裡 → Agent 轉移
├── 流程固定，只是步驟多 → Workflow
├── 主 Agent 需要一個小幫手 → AgentTool
└── 混合場景 → Workflow + Agent 節點
```

## 模式 6：依賴反轉（Dependency Inversion）

### 應用位置
所有 Service（Session、Memory、Artifact）

### 核心概念
```python
# 依賴抽象，不依賴實作
agent 依賴 → BaseSessionService（抽象）
                    ↑ 實作
            InMemorySessionService（開發）
            DatabaseSessionService（生產）
```

### 何時使用
- 需要在不同環境用不同實作
- 測試時需要 Mock
- 避免程式碼和基礎設施耦合

### ADK 中的應用
```python
# 開發時
runner = Runner(session_service=InMemorySessionService())

# 生產時（只改這一行）
runner = Runner(session_service=DatabaseSessionService(db_url))

# Agent 程式碼完全不變！
```

## 模式 7：策略模式（Strategy）

### 應用位置
Flow 的自動選擇

### 核心概念
```python
# 根據條件選擇不同的演算法
if has_sub_agents:
    strategy = AutoFlow   # 支援轉移
else:
    strategy = SingleFlow  # 單純對話
```

### ADK 中的應用
```python
class LlmAgent:
    @property
    def _llm_flow(self):
        if self.sub_agents:
            return AutoFlow()
        return SingleFlow()
```

## 模式 8：AsyncGenerator 串流

### 應用位置
所有需要即時回應的地方

### 核心概念
```python
async def generate() -> AsyncGenerator[Event, None]:
    yield partial_event_1  # 即時發送
    yield partial_event_2  # 即時發送
    yield final_event      # 完成
```

### 為什麼不用 Callback？

| AsyncGenerator | Callback |
|---------------|----------|
| 呼叫者控制流程 | 被呼叫者控制 |
| 可以 break | 需要取消機制 |
| 自然背壓 | 需要手動控制 |
| 可組合 | 難組合 |

## 模式 9：惰性載入（Lazy Loading）

### 應用位置
Tool 模組、Model 供應商

### ADK 中的應用
```python
# tools/__init__.py 用 __getattr__ 實現惰性載入
_MODULES = {
    "MCPToolset": "google.adk.tools.mcp_tool",
    "OpenAPIToolset": "google.adk.tools.openapi_tool",
    ...
}

def __getattr__(name):
    if name in _MODULES:
        module = importlib.import_module(_MODULES[name])
        return getattr(module, name)
    raise AttributeError(name)
```

**效果：** 只有在你實際使用 `MCPToolset` 時才會載入 MCP 相關的依賴。如果你不用 MCP，那些套件不需要安裝。

## 模式 10：介面統一（Interface Unification）

### 應用位置
BaseAgent 繼承 BaseNode

### 核心概念
```
讓不同用途的元件共享同一個介面
→ Agent 可以獨立運行，也可以當 Workflow 節點
→ 不需要 Adapter，直接相容
```

### 何時使用
- 元件會在多個 context 中使用
- 想避免 Adapter 的複雜性
- 提前規劃系統的擴展方向

## 綜合架構決策指南

### 選擇你的 Agent 架構

```
Q1: 你的 Agent 有幾個？
├── 一個 → 單 LlmAgent，配好工具就行
└── 多個 → Q2

Q2: 它們之間的關係？
├── 固定順序 → Workflow（線性邊）
├── 條件分支 → Workflow（路由邊）
├── 動態決定 → Agent 轉移（sub_agents）
└── 獨立子任務 → AgentTool

Q3: 需要什麼級別的可靠性？
├── 開發階段 → InMemory 一切
├── 小規模 → SQLite + 本地
└── 生產 → Database/Vertex + 重試 + 監控

Q4: 怎麼選模型？
├── 簡單任務（分類/路由）→ Flash/Haiku
├── 一般任務 → Pro/Sonnet
├── 複雜推理 → Pro/Opus
└── 混合 → 根 Agent 用貴的，子 Agent 用便宜的
```

## 反模式（要避免的）

| 反模式 | 問題 | 正確做法 |
|--------|------|----------|
| 超級大 Agent | 指令太長，工具太多，容易搞混 | 拆分成多個專業 Agent |
| 所有東西放 State | State 爆炸 | 只放必要的，用 Memory 存長期資訊 |
| 過度使用回呼 | 邏輯分散，難追蹤 | 回呼只做橫切關注點（日誌、監控） |
| 忽略串流 | 用戶體驗差 | 總是用 AsyncGenerator |
| 硬編碼模型 | 無法切換供應商 | 用字串名稱 + Registry |
| 跳過評估 | 不知道 Agent 品質 | 建立 eval set，持續評估 |
