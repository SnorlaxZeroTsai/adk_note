## 名詞速查表

首次閱讀時可能會遇到不熟悉的術語，這裡快速解釋：

### 基礎概念

| 術語 | 一句話解釋 |
|------|-----------|
| Token | LLM 處理文字的最小單位，約等於一個英文單字或 1-2 個中文字 |
| Provider | 提供 LLM API 的供應商（如 OpenAI、Anthropic、Google） |
| Completion | 向 LLM 發送一段文字，取得它的回覆。這個動作叫 "completion" |
| Streaming | 不等 LLM 全部回完，邊生成邊接收（像打字一樣逐字出現） |
| Context Window | LLM 一次能處理的最大 token 數（記憶長度） |
| Proxy / Gateway | 站在你和供應商之間的中間伺服器，負責路由、認證、限流 |

### 路由與容錯

| 術語 | 一句話解釋 |
|------|-----------|
| Router | 決定一個請求應該發到哪個供應商 / 哪個 API Key 的元件 |
| Deployment | 一個可實際呼叫的 LLM 端點（Provider + API Key + endpoint） |
| Fallback | 主要的供應商失敗後，自動切換到備用的機制 |
| Cooldown | 部署失敗率過高時，暫時停止向它發送請求的機制 |
| Thompson Sampling | 多臂吃角子老虎機演算法，在探索與利用之間取得最佳平衡 |

### 快取

| 術語 | 一句話解釋 |
|------|-----------|
| LRU Cache | Least Recently Used 快取，滿了就淘汰最久沒用的那筆 |
| TTL | Time To Live，快取條目的存活時間，過期自動失效 |
| DualCache | LiteLLM 的雙層快取（Redis + InMemory），兼顧一致性與低延遲 |

### 網路通訊

| 術語 | 一句話解釋 |
|------|-----------|
| SSE | Server-Sent Events，伺服器推送事件的協議（用於 LLM streaming） |
| HTTP/2 多路複用 | 一條 TCP 連線同時傳輸多個請求/回應，避免 HTTP/1.1 隊頭阻塞 |
| 連線池 | 預先建立 TCP 連線重複使用，避免每次請求都建新連線的開銷 |
| 背壓（Backpressure） | 下游消費速度跟不上上游生產速度時的流量控制機制 |

### 安全與認證

| 術語 | 一句話解釋 |
|------|-----------|
| API Key | 一串隨機字串，作為身份證明（像密碼但更長更隨機） |
| JWT | JSON Web Token，自帶身份資訊的令牌，可離線驗證 |
| OAuth2 | 委託第三方（如 Google）驗證使用者身份的協議 |
| RBAC | Role-Based Access Control，按角色分配權限 |
| Guardrail | 安全護欄——在 LLM 呼叫前中後攔截審查的安全層 |
| PII | Personally Identifiable Information，個人識別資訊（姓名、信用卡號等） |
| JWKS | JSON Web Key Set，公鑰的公開目錄，用於驗證 JWT 簽名 |

### 其他

| 術語 | 一句話解釋 |
|------|-----------|
| MCP | Model Context Protocol，讓 AI 助手調用外部工具的標準協議 |
| RPM / TPM | Requests Per Minute / Tokens Per Minute，速率限制的兩種維度 |
| Virtual Key | LiteLLM Proxy 自己發的虛擬 key，使用者不接觸真正的 Provider key |

## 環境準備（可選）

如果你想邊讀邊跑程式碼：

```bash
# 1. Clone LiteLLM 原始碼
git clone https://github.com/BerriAI/litellm.git
cd litellm

# 2. 建立虛擬環境
python -m venv .venv && source .venv/bin/activate

# 3. 安裝開發依賴
pip install -e ".[dev]"

# 4. 確認安裝成功
python -c "import litellm; print(litellm.__version__)"
```
