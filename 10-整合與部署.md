# 10 - 整合與部署

## ADK 的整合生態系

ADK 不是一個封閉系統，它提供多種方式和外部世界連接：

```
                    ┌─────────────┐
                    │   你的 Agent  │
                    └──────┬──────┘
                           │
        ┌──────────────────┼──────────────────┐
        │                  │                  │
    ┌───▼────┐      ┌─────▼─────┐     ┌─────▼─────┐
    │  MCP   │      │  OpenAPI  │     │   A2A     │
    │ 伺服器  │      │  規格    │     │  協定     │
    └────────┘      └───────────┘     └───────────┘
    外部工具伺服器    REST API 自動整合   Agent 對 Agent
```

## MCP 整合（Model Context Protocol）

### MCP 是什麼？

MCP 是一個開放協定，讓 LLM 應用可以連接到外部工具伺服器。想像它是 Agent 世界的「USB」——標準化的介面。

**檔案位置：** `src/google/adk/tools/mcp_tool/`

### 使用方式

```python
from google.adk.tools.mcp_tool import MCPToolset, StdioConnectionParams

# 連接到一個 MCP 伺服器（例如：檔案系統操作）
mcp_tools = MCPToolset(
    connection_params=StdioConnectionParams(
        command="npx",
        args=["-y", "@modelcontextprotocol/server-filesystem", "/tmp"]
    ),
)

agent = Agent(
    name="file_agent",
    tools=[mcp_tools],  # 自動獲得 MCP 伺服器提供的所有工具
)
```

### 三種連接方式

| 方式 | 適用場景 | 說明 |
|------|----------|------|
| `StdioConnectionParams` | 本地程序 | 啟動子程序，透過 stdin/stdout 通訊 |
| `SseConnectionParams` | 遠端伺服器 | 透過 Server-Sent Events 連接 |
| `StreamableHTTPConnectionParams` | HTTP 串流 | 透過 HTTP 串流連接 |

### MCPToolset 的內部運作

```
MCPToolset.get_tools()
    │
    ▼
MCPSessionManager 連接到 MCP 伺服器
    │
    ▼
取得伺服器提供的工具列表
    │
    ▼
每個 MCP 工具 → 包裝成 MCPTool（繼承 BaseTool）
    │
    ▼
MCPTool.run_async() 時，透過 MCP 協定呼叫伺服器
```

**資深工程師觀點：** MCP 讓你不需要為每個外部工具寫 Python 包裝器。有人寫好 MCP 伺服器，你直接連就能用。這是「買 vs 做」的典型案例。

## OpenAPI 整合

### 自動從 API 規格產生工具

```python
from google.adk.tools.openapi_tool import OpenAPIToolset

# 載入 OpenAPI 規格（可以是 YAML 或 JSON）
toolset = OpenAPIToolset(
    spec=open("api_spec.yaml").read(),
    # 只暴露這些端點
    tool_filter=["getUser", "createOrder", "searchProducts"],
    # 認證設定
    auth_scheme="bearer",
    auth_credential="your-api-token",
)

agent = Agent(
    name="api_agent",
    tools=[toolset],
)
```

### 它做了什麼？

```
OpenAPI YAML/JSON
    │
    ▼
OpenApiSpecParser 解析規格
    │
    ▼
每個 API Operation → RestApiTool
    │  - GET /users/{id} → get_user 工具
    │  - POST /orders → create_order 工具
    │  - GET /products?q= → search_products 工具
    ▼
RestApiTool.run_async() 時，發送實際的 HTTP 請求
```

### 認證處理

```python
# 支援多種認證方式：
OpenAPIToolset(
    spec=spec,
    auth_scheme="oauth2",      # OAuth2
    # 或
    auth_scheme="apiKey",      # API Key
    # 或
    auth_scheme="bearer",      # Bearer Token
)
```

**資深工程師觀點：** 如果你的公司有很多微服務，每個都有 OpenAPI 規格，你可以快速讓 Agent 操作所有服務，不需要為每個 API 手動寫工具。

## A2A 整合（Agent-to-Agent Protocol）

### A2A 是什麼？

A2A 是讓 **不同框架** 的 Agent 互相溝通的協定。你的 ADK Agent 可以和用其他框架（如 LangChain、AutoGen）建構的 Agent 對話。

**檔案位置：** `src/google/adk/a2a/`

### 使用場景

```
┌─────────────┐         A2A 協定         ┌─────────────┐
│  你的 ADK   │  ←──────────────────────→ │  別人的      │
│  Agent      │                           │  LangChain  │
│             │                           │  Agent      │
└─────────────┘                           └─────────────┘
```

### 核心元件

```python
# A2aAgentExecutor：把 ADK Agent 暴露為 A2A 服務
from google.adk.a2a.executor import A2aAgentExecutor

executor = A2aAgentExecutor(
    runner=runner,
    agent=my_agent,
)
# 現在其他框架的 Agent 可以透過 A2A 協定呼叫你的 Agent
```

**注意：** A2A 在 ADK 中仍是實驗性功能（標記 `@a2a_experimental`）。

## CLI 系統

**檔案位置：** `src/google/adk/cli/`

### 可用命令

```bash
# 互動式對話
adk run path/to/my_agent

# 啟動 Web 開發界面
adk web path/to/agents_dir

# 部署到雲端
adk deploy cloud_run path/to/my_agent
adk deploy agent_engine path/to/my_agent

# 執行評估
adk eval path/to/my_agent --eval_set test_cases.json

# 建立新 Agent 專案
adk create my_new_agent
```

### Web 開發界面（adk web）

```
啟動後你得到：
├── 對話界面（即時對話測試）
├── Events 頁面（查看完整執行軌跡）
├── Graph 視覺化（Workflow 的圖形化呈現）
├── Eval 介面（執行和查看評估結果）
└── Debug 工具（檢查 Session、State）
```

**架構：**
- 後端：FastAPI 伺服器
- 前端：Angular 應用
- 通訊：SSE（Server-Sent Events）做串流

### 程式化 API 伺服器

```python
from google.adk.cli.fast_api import get_fast_api_app

# 一行程式碼把 Agent 變成 API
app = get_fast_api_app(agent_dir="./agents")

# 加自訂端點
@app.get("/health")
async def health():
    return {"status": "ok"}

# 用 uvicorn 啟動
# uvicorn main:app --host 0.0.0.0 --port 8000
```

### 標準 API 端點

| 端點 | 用途 |
|------|------|
| `POST /run` | 執行 Agent（非串流） |
| `POST /run_sse` | 執行 Agent（SSE 串流） |
| `POST /run_live` | 即時雙向（WebSocket） |
| `GET /list-apps` | 列出可用 Agent |
| `GET /apps/{name}/sessions` | 取得 Session 列表 |

## 部署策略

### 1. Google Cloud Run

```bash
adk deploy cloud_run ./my_agent \
    --project my-gcp-project \
    --region us-central1
```

**適合：** 無狀態、自動擴展、按使用量計費

### 2. Google Vertex Agent Engine

```bash
adk deploy agent_engine ./my_agent \
    --project my-gcp-project \
    --region us-central1
```

**適合：** 需要 Google 託管的 Session、Memory、Artifact 服務

### 3. Google GKE（Kubernetes）

```bash
adk deploy gke ./my_agent \
    --cluster my-cluster \
    --namespace production
```

**適合：** 需要完全控制基礎設施、已有 K8s 叢集

### 4. 自行部署（FastAPI）

```python
# 最大彈性：自己跑 FastAPI
app = get_fast_api_app(agent_dir="./agents")
# 部署到任何支援 Python 的平台
```

## 設計洞察

### 1. 為什麼支援這麼多整合方式？

因為真實世界的 Agent 需要：
- 呼叫現有 API（OpenAPI）
- 連接現有工具生態系（MCP）
- 和其他 Agent 系統互動（A2A）
- 部署到不同環境

**沒有銀彈，每種整合解決不同問題。**

### 2. CLI 和 API 的關係

```
CLI（開發體驗）
    │
    ▼ 底層呼叫
FastAPI App（API 層）
    │
    ▼ 底層呼叫
Runner（執行引擎）
    │
    ▼ 底層呼叫
Agent + Flow + Tools
```

CLI 只是 Runner 的一個「皮」，和你自己寫的 API 呼叫一模一樣的底層邏輯。

### 3. 部署的關鍵考量

| 考量 | 選擇 |
|------|------|
| 需要持久 Session？ | Database SessionService + 有狀態部署 |
| 需要水平擴展？ | Cloud Run + 外部狀態儲存 |
| 需要即時串流？ | SSE 或 WebSocket 支援 |
| 需要長時間任務？ | GKE + 長連線支援 |
| 開發中？ | `adk web`（零配置） |
