## 從 LiteLLM 學到的系統設計智慧

研讀 LiteLLM 的源碼不僅僅是學一個工具，更是學習如何設計大型 AI 基礎設施。以下是你可以帶走的核心設計模式和思維。

## 一、設計模式精粹

### 1. 適配器模式（Adapter Pattern）

**LiteLLM 中的應用**：每個 Provider Config 都是一個適配器。

```python
# 統一介面
class BaseConfig(ABC):
    @abstractmethod
    def transform_request(self, ...): ...
    @abstractmethod
    def transform_response(self, ...): ...

# 適配器實作
class AnthropicConfig(BaseConfig): ...
class GeminiConfig(BaseConfig): ...
```

**何時使用**：當你需要統一多個不同系統的介面時。在 AI 領域，這幾乎是日常——不同的模型服務、向量資料庫、embedding API 都有不同的格式。

### 2. 策略模式（Strategy Pattern）

**LiteLLM 中的應用**：路由策略。

```python
# 策略介面
class RoutingStrategy:
    def pick_deployment(self, healthy_deployments): ...

# 具體策略
class SimpleShuffleStrategy: ...
class LowestLatencyStrategy: ...
class CostBasedStrategy: ...
```

**何時使用**：當同一個操作有多種可互換的演算法時。AI 系統中常見於：模型選擇策略、RAG 檢索策略、prompt 選擇策略。

### 3. 裝飾器模式（Decorator Pattern）

**LiteLLM 中的應用**：`@client` 裝飾器。

```python
@client  # 加入 logging、caching、cost tracking
def completion(model, messages, ...):
    # 核心邏輯
    ...
```

**何時使用**：需要在不修改核心邏輯的情況下加入橫切關注點（日誌、快取、監控）。

### 4. 觀察者模式（Observer Pattern）

**LiteLLM 中的應用**：Callback 系統。

```python
litellm.success_callback = [langfuse, prometheus, custom_logger]
# 每次成功呼叫，所有觀察者都會被通知
```

**何時使用**：當一個事件需要通知多個不同的系統時。AI 可觀測性是經典場景。

## 二、AI 工程核心概念

### Token 經濟學

理解 token 是 AI 工程的基礎：

```
1 token ≈ 4 個英文字元 ≈ 0.75 個英文單字
1 token ≈ 1-2 個中文字元

GPT-4o 定價（2024）：
- Input:  $2.50 / 1M tokens
- Output: $10.00 / 1M tokens
- Cached: $1.25 / 1M tokens（50% 折扣）

Claude 3.5 Sonnet 定價：
- Input:  $3.00 / 1M tokens
- Output: $15.00 / 1M tokens
- Cache write: $3.75 / 1M tokens
- Cache read:  $0.30 / 1M tokens（90% 折扣！）
```

**核心洞察**：
- Output token 比 input token 貴 3-5 倍
- 快取是成本優化的最大槓桿
- 一次呼叫的成本 = f(模型選擇, input 長度, output 長度, 是否快取)

### Context Window 管理

```
┌────────────────────────────────────────┐
│           Context Window               │
│                                        │
│  ┌──────────┐ ┌──────────┐ ┌───────┐  │
│  │ System   │ │ History  │ │ Query │  │
│  │ Prompt   │ │ Messages │ │       │  │
│  │ (固定)   │ │ (動態)   │ │(變動) │  │
│  └──────────┘ └──────────┘ └───────┘  │
│                                        │
│  ← 越長越貴，但也越有上下文 →           │
└────────────────────────────────────────┘
```

**策略**：
1. **滑動視窗**：只保留最近 N 條歷史
2. **摘要壓縮**：將舊歷史壓縮為摘要
3. **RAG 增強**：只在需要時檢索相關上下文
4. **分層快取**：利用供應商的 prompt caching

### 負載均衡的藝術

AI 系統的負載均衡與傳統 Web 不同：

```
傳統 Web：
- 請求短且均勻（~100ms）
- 無狀態
- CPU/Memory bound

AI 系統：
- 請求長且不均勻（100ms ~ 120s）
- Token 限制而非請求限制
- GPU bound，排隊嚴重
- Rate limit 是硬性約束
```

**LiteLLM 的解法**：
- 追蹤 TPM/RPM 而非 QPS
- 冷卻機制處理暫時性故障
- 延遲感知路由避免慢節點
- 權重配置處理異質部署

### 錯誤處理的層次

```
┌─────────────────────────────────────┐
│ 層次 1：參數錯誤（可預防）           │
│ → 驗證 + drop_params               │
├─────────────────────────────────────┤
│ 層次 2：暫時性錯誤（可重試）         │
│ → 429 Rate Limit → 指數退避重試    │
│ → 500 Server Error → 換一個 deployment │
├─────────────────────────────────────┤
│ 層次 3：永久性錯誤（需 fallback）    │
│ → Context too long → 換更大的模型   │
│ → Content filter → 換其他供應商     │
├─────────────────────────────────────┤
│ 層次 4：系統級錯誤（需告警）         │
│ → 所有 deployment 都失敗            │
│ → 預算用完                         │
└─────────────────────────────────────┘
```

## 五、從源碼閱讀中獲得的洞察

### 1. 好的抽象比好的程式碼更重要

LiteLLM 的 `BaseConfig` 抽象讓 100+ 供應商的整合變得可管理。一個好的抽象可以讓系統在不修改核心的情況下擴展。

### 2. 全域狀態有時是務實的選擇

`litellm.success_callback = [...]` 這種全域狀態設計被許多人批評，但它確實讓使用者的程式碼極其簡潔。工程是取捨的藝術。

### 3. 遷移是持續的過程

LiteLLM 從「每個供應商一個 handler」遷移到「統一 BaseLLMHTTPHandler + Config」。在真實專案中，你幾乎永遠無法一次性重構完成。

### 4. 可觀測性不是事後的想法

從一開始就嵌入 logging、cost tracking、callbacks——這些不是「nice to have」，是生產環境的必需品。

### 5. 簡單性 > 完美性

`simple_shuffle` 路由策略就是一個 `random.choice()`。對大多數場景來說，這就夠了。不要過度工程化。

## 給初學者的最後建議

1. **不要試圖一次理解所有東西**。先跑通一個最簡單的 `litellm.completion()` 呼叫，再逐層深入。
2. **讀源碼時帶著問題**。「這個函式為什麼在這裡？」比「我要讀完所有檔案」有效 100 倍。
3. **貢獻不一定要寫程式碼**。改善文件、回報 bug、回答社群問題都是有價值的貢獻。
4. **建立自己的筆記**。把「我最初不理解但後來搞懂」的東西記下來——這就是最好的教學材料。
5. **AI 領域變化極快**。今天學的具體 API 明天可能就變了，但設計模式和系統思維是持久的。
