## 一、設計模式深度剖析

### 1. 適配器模式（Adapter Pattern）

#### LiteLLM 的做法

每個 Provider 是一個 Adapter，實作統一介面：

```python
class BaseConfig(ABC):
    @abstractmethod
    def transform_request(
        self, model: str, messages: list, optional_params: dict, headers: dict
    ) -> dict:
        """將統一格式轉成 Provider 特定格式"""
        ...

    @abstractmethod
    def transform_response(
        self, model: str, raw_response: httpx.Response, logging_obj: Any
    ) -> ModelResponse:
        """將 Provider 回應轉成統一格式"""
        ...

class AnthropicConfig(BaseConfig):
    def transform_request(self, model, messages, optional_params, headers):
        # OpenAI 格式 → Anthropic 格式
        return {
            "model": model,
            "messages": self._convert_messages(messages),  # role 映射不同
            "max_tokens": optional_params.get("max_tokens", 4096),
            "system": self._extract_system_prompt(messages),
        }

    def transform_response(self, model, raw_response, logging_obj):
        # Anthropic 回應 → 統一 ModelResponse
        json_resp = raw_response.json()
        return ModelResponse(
            choices=[Choice(message=Message(
                content=json_resp["content"][0]["text"],
                role="assistant",
            ))],
            usage=Usage(
                prompt_tokens=json_resp["usage"]["input_tokens"],
                completion_tokens=json_resp["usage"]["output_tokens"],
            ),
        )
```

#### 為什麼不用繼承（Inheritance）？

**替代方案：Template Method Pattern**

```python
# ❌ 用繼承的做法
class BaseLLMProvider:
    def completion(self, messages):
        validated = self.validate(messages)        # 步驟 1
        request = self.build_request(validated)    # 步驟 2（子類覆寫）
        response = self.send(request)             # 步驟 3
        return self.parse_response(response)      # 步驟 4（子類覆寫）

class OpenAIProvider(BaseLLMProvider):
    def build_request(self, messages): ...
    def parse_response(self, response): ...

class AnthropicProvider(BaseLLMProvider):
    def build_request(self, messages): ...
    def parse_response(self, response): ...
```

**為什麼 LiteLLM 沒這樣做？**

| 問題 | 繼承 | 適配器（組合） |
|------|------|--------------|
| Provider 需要不同的 HTTP 行為 | 只能覆寫固定步驟 | Config 和 Handler 獨立變化 |
| 新增一個 hook 點 | 改 base class → 影響所有子類 | 加一個新方法，不影響現有 Config |
| 一個 Provider 支援多格式 | 需要多重繼承或 mixin | 一個 Provider 多個 Config |
| 測試 | 需要 mock 整個繼承鏈 | 可以獨立測試 Config |

**真實案例**：Azure OpenAI 跟 OpenAI 的 API 幾乎一樣，只是 URL 和 auth 不同。用繼承的話要做 `AzureOpenAI(OpenAIProvider)` 然後覆寫一堆東西。用適配器的話：

```python
# LiteLLM 的做法：AzureConfig 只覆寫 URL/Auth 相關的方法
class AzureOpenAIConfig(OpenAIConfig):
    def get_complete_url(self, endpoint, model, ...):
        # Azure 的 URL 格式不同
        return f"{endpoint}/openai/deployments/{model}/chat/completions?api-version={api_version}"

    def validate_environment(self, headers, model, ...):
        # Azure 用 api-key header 而非 Bearer token
        headers["api-key"] = api_key
        return headers
```

只改需要改的，其他都繼承 OpenAI 的行為。

#### 為什麼不用 Factory + if/else？

```python
# ❌ 最原始的做法
def completion(model, messages):
    if model.startswith("gpt"):
        # 200 行 OpenAI 邏輯
        ...
    elif model.startswith("claude"):
        # 200 行 Anthropic 邏輯
        ...
    elif model.startswith("gemini"):
        # 200 行 Google 邏輯
        ...
    # 100+ 個 provider...
```

**為什麼不行？**
- 一個 `completion()` 函式會膨脹到上萬行
- 加新 Provider 要改核心程式碼（違反 Open-Closed Principle）
- 無法獨立測試每個 Provider
- LiteLLM 有 100+ Provider，if/else 完全無法維護

**適配器的優勢**：新增 Provider = 新增一個 Config 檔案，不碰核心邏輯。

### 2. 策略模式（Strategy Pattern）

#### LiteLLM 的路由策略

```python
# Router 支援多種路由策略
class Router:
    def __init__(self, routing_strategy: str = "simple-shuffle"):
        self.routing_strategy = routing_strategy

    def get_available_deployment(self, model, healthy_deployments):
        if self.routing_strategy == "simple-shuffle":
            return random.choice(healthy_deployments)
        elif self.routing_strategy == "least-busy":
            return min(healthy_deployments, key=lambda d: d.active_requests)
        elif self.routing_strategy == "latency-based-routing":
            return min(healthy_deployments, key=lambda d: d.avg_latency)
        elif self.routing_strategy == "cost-based-routing":
            return min(healthy_deployments, key=lambda d: d.cost_per_token)
```

#### 為什麼不用 Chain of Responsibility？

**替代方案**：讓多個 handler 依序嘗試

```python
# ❌ Chain of Responsibility
class RoutingHandler:
    def __init__(self, next_handler=None):
        self.next = next_handler

    def handle(self, request, deployments):
        result = self.try_route(request, deployments)
        if result:
            return result
        if self.next:
            return self.next.handle(request, deployments)
        raise NoDeploymentAvailable()

class CostHandler(RoutingHandler):
    def try_route(self, request, deployments):
        cheap = [d for d in deployments if d.cost < threshold]
        return random.choice(cheap) if cheap else None

class LatencyHandler(RoutingHandler):
    def try_route(self, request, deployments):
        fast = [d for d in deployments if d.latency < threshold]
        return random.choice(fast) if fast else None

# 使用：cost → latency → fallback
chain = CostHandler(LatencyHandler(FallbackHandler()))
```

**為什麼不適合路由場景？**

| 問題 | Chain of Responsibility | Strategy |
|------|------------------------|----------|
| 語意 | 「誰能處理這個請求」（過濾） | 「用什麼演算法選」（選擇） |
| 路由本質 | 是選擇問題，不是過濾問題 | ✅ 正確語意 |
| 可預測性 | 不知道哪個 handler 會生效 | 明確知道用什麼策略 |
| 除錯 | 要追蹤整條 chain | 直接看策略名稱 |
| 配置 | 組裝 chain 的順序影響結果 | 一個字串就決定 |

**關鍵洞察**：路由的需求是「從 N 個健康部署中選一個」，不是「找到第一個能處理的人」。Strategy 的語意更精確。

#### 為什麼不用 State Machine？

```python
# ❌ 用狀態機做路由
class RouterState:
    NORMAL = "normal"
    DEGRADED = "degraded"  # 部分部署掛了
    EMERGENCY = "emergency"  # 所有主要都掛了

class StatefulRouter:
    def __init__(self):
        self.state = RouterState.NORMAL

    def route(self, deployments):
        if self.state == RouterState.NORMAL:
            return self.normal_routing(deployments)
        elif self.state == RouterState.DEGRADED:
            return self.degraded_routing(deployments)  # 避免有問題的
        elif self.state == RouterState.EMERGENCY:
            return self.emergency_routing(deployments)  # fallback provider
```

**為什麼 LiteLLM 不用狀態機？**

因為 LiteLLM 已經有 **Cooldown Cache** 做這件事了！

```
狀態機思路：Router 有全域狀態 → 根據狀態選策略
LiteLLM 思路：每個 Deployment 獨立有狀態（cooldown/healthy）→ 過濾掉不健康的 → 對剩餘的用策略選

LiteLLM 的做法更好因為：
1. 部分故障是精確到單個 deployment 的，不是全域狀態
2. 不需要手動管理狀態轉換（cooldown 自動 TTL 過期）
3. 新增部署不需要修改狀態機邏輯
```

### 3. 觀察者模式（Observer Pattern）

#### LiteLLM 的 Callback 系統

```python
# 設定多個觀察者
litellm.success_callback = ["langfuse", "prometheus", "custom_logger"]
litellm.failure_callback = ["alerting", "sentry"]

# 每次 completion 成功/失敗時，自動通知所有觀察者
# 內部實作：
class Logging:
    async def async_success_handler(self, result, start_time, end_time):
        for callback in litellm.success_callback:
            try:
                await callback.async_log_success_event(
                    kwargs=self.model_call_details,
                    response_obj=result,
                    start_time=start_time,
                    end_time=end_time,
                )
            except Exception:
                pass  # 一個 callback 失敗不影響其他
```

#### 為什麼不用 Middleware / Pipeline？

**替代方案：Middleware Chain（像 Express.js / Django）**

```python
# ❌ Middleware 方式
class LoggingMiddleware:
    async def __call__(self, request, next):
        start = time.time()
        response = await next(request)  # 呼叫下一個 middleware
        duration = time.time() - start
        await self.log(request, response, duration)
        return response

class CacheMiddleware:
    async def __call__(self, request, next):
        cached = await self.cache.get(request)
        if cached:
            return cached
        response = await next(request)
        await self.cache.set(request, response)
        return response

# 組裝
pipeline = LoggingMiddleware(CacheMiddleware(ActualLLMCall()))
```

**兩者的差異：**

| 特性 | Observer (Callback) | Middleware (Pipeline) |
|------|--------------------|-----------------------|
| 執行時機 | 事後通知（fire-and-forget） | 請求前後（可攔截） |
| 能修改結果嗎 | ❌ 不行 | ✅ 可以修改 request/response |
| 順序重要嗎 | 不重要（互相獨立） | 重要（外到內再到外） |
| 一個失敗的影響 | 不影響其他 | 可能中斷整條 chain |
| 適合場景 | 日誌、metrics、通知 | 認證、快取、轉換 |

**LiteLLM 為什麼選 Observer？**

Callback 的語意是「通知」，不是「處理」。Langfuse、Prometheus、自定義 logger 這些觀察者：
- 不需要修改 LLM 的回應
- 彼此之間不需要知道對方存在
- 一個掛了不應該影響主流程
- 使用者要能隨時加減

如果用 Middleware，一個 Prometheus exporter 的 bug 就可能讓整條 pipeline 掛掉。

**但 LiteLLM 也有用 Middleware 的地方**：

```python
# Proxy 的 pre-call hooks（需要攔截/修改）
class MaxBudgetLimiter:  # 這是 middleware 語意
    async def async_pre_call_hook(self, data, user_api_key_dict):
        if user_api_key_dict.spend >= user_api_key_dict.max_budget:
            raise BudgetExceededError(...)  # 攔截請求！
        return data  # 可以修改 data
```

**結論**：Observer 用於「事後通知」（日誌、metrics），Middleware 用於「可以攔截或修改請求」（認證、預算檢查）。兩者共存，各司其職。

#### 為什麼不用 Event Bus / Pub-Sub？

```python
# ❌ Event Bus 方式
class EventBus:
    def __init__(self):
        self._subscribers = defaultdict(list)

    def subscribe(self, event_type: str, handler: Callable):
        self._subscribers[event_type].append(handler)

    async def publish(self, event_type: str, data: dict):
        for handler in self._subscribers[event_type]:
            await handler(data)

# 使用
bus = EventBus()
bus.subscribe("completion.success", langfuse_handler)
bus.subscribe("completion.success", prometheus_handler)
bus.subscribe("completion.failure", sentry_handler)
```

**這看起來更「正規」，為什麼 LiteLLM 沒用？**

1. **簡單性**：`litellm.success_callback = [...]` 一行搞定，Event Bus 需要額外的基礎設施
2. **使用者認知成本**：list 人人都懂，Event Bus 需要學習 API
3. **事件類型膨脹**：Event Bus 鼓勵創造大量事件類型（`completion.success`, `completion.stream.chunk`, `completion.stream.end`, ...），增加維護負擔
4. **LiteLLM 的 callback 不需要 pub-sub 的優勢**：沒有跨進程需求、沒有事件持久化需求、沒有事件重播需求

**什麼時候該用 Event Bus？** 當系統跨多個服務、需要事件持久化（Kafka）、或事件類型會動態增長時。LiteLLM 是單進程 SDK，Callback list 就夠了。

### 4. 裝飾器模式（Decorator Pattern）

#### LiteLLM 的 `@client` 裝飾器

```python
@client
async def acompletion(model, messages, **kwargs):
    """核心邏輯只做一件事：呼叫 Provider"""
    response = await handler.async_completion(model, messages, kwargs)
    return response

# @client 偷偷做了這些事（簡化版）：
def client(original_function):
    @wraps(original_function)
    async def wrapper(*args, **kwargs):
        # 1. 開始計時
        start_time = time.time()

        # 2. 檢查快取
        cached = check_cache(kwargs)
        if cached:
            return cached

        # 3. 執行核心邏輯
        try:
            result = await original_function(*args, **kwargs)
        except Exception as e:
            # 4. 失敗回呼
            await failure_callback(e, start_time)
            raise

        # 5. 成功回呼（日誌、metrics、cost tracking）
        await success_callback(result, start_time)

        # 6. 寫入快取
        set_cache(kwargs, result)

        return result
    return wrapper
```

#### 為什麼不用 AOP（Aspect-Oriented Programming）？

**替代方案：AOP 框架**

```python
# ❌ AOP 方式（像 Java 的 Spring AOP）
@aspect
class LoggingAspect:
    @before("execution(* completion(..))")
    def before_completion(self, join_point):
        self.start_time = time.time()

    @after_returning("execution(* completion(..))")
    def after_completion(self, join_point, result):
        duration = time.time() - self.start_time
        log_success(result, duration)

    @after_throwing("execution(* completion(..))")
    def on_error(self, join_point, exception):
        log_failure(exception)
```

**為什麼 Python 不用 AOP？**

| 問題 | AOP | Python Decorator |
|------|-----|-----------------|
| 語言支援 | Java 有 AspectJ | Python 原生支援 decorator |
| 隱式 vs 顯式 | 隱式攔截（看不到在哪裡生效） | 顯式標記（`@client` 清楚可見） |
| 除錯 | 難追蹤（magic proxy） | 正常的 call stack |
| 靈活性 | 強大但重量級 | 輕量，夠用就好 |

Python 的 decorator 已經是「窮人版 AOP」——足夠處理橫切關注點，又不引入額外框架。

#### 為什麼不把這些邏輯直接寫在函式裡？

```python
# ❌ 不用裝飾器，全部塞在一起
async def acompletion(model, messages, **kwargs):
    start_time = time.time()

    # 快取檢查（20 行）
    cached = check_cache(...)
    if cached:
        return cached

    try:
        # 核心邏輯（5 行）
        response = await handler.async_completion(model, messages, kwargs)
    except Exception as e:
        # 錯誤處理（30 行）
        await failure_callback(e, start_time)
        raise

    # 成功後處理（40 行）
    await success_callback(response, start_time)
    set_cache(kwargs, response)

    return response
```

**看起來也行，為什麼不好？**

因為 LiteLLM 有多個入口函式都需要同樣的橫切邏輯：

```python
@client
async def acompletion(...): ...      # 文字生成

@client
async def aembedding(...): ...       # Embedding

@client
async def aimage_generation(...): ... # 圖片生成

@client
async def atranscription(...): ...    # 語音轉文字
```

如果不用 decorator，每個函式都要重複寫一遍快取/日誌/metrics 邏輯。**DRY 原則**。

### 5. 代理模式（Proxy Pattern）— LiteLLM Gateway

#### 為什麼整個 Gateway 本身就是 Proxy Pattern？

```
Client App
    │
    ▼
LiteLLM Proxy（代理）
    │   - 認證
    │   - 限流
    │   - 預算檢查
    │   - 路由
    │   - 日誌
    ▼
Real LLM Provider
```

Client 以為自己在跟 OpenAI API 說話（相同的介面），但中間的 Proxy 加了大量控制邏輯。

#### 為什麼不用 Sidecar Pattern（Service Mesh 方式）？

**替代方案：像 Istio/Envoy 那樣用 Sidecar**

```
┌─────────────────────────┐
│ Pod                      │
│ ┌─────────┐ ┌─────────┐ │
│ │ Your App│→│ Sidecar │─┼──→ LLM Provider
│ │         │ │ (Envoy) │ │
│ └─────────┘ └─────────┘ │
└─────────────────────────┘
```

Sidecar 攔截所有出站流量，自動做 retry、circuit breaker、metrics。

**為什麼 LiteLLM 不是 Sidecar？**

| 考量 | Sidecar | LiteLLM Proxy |
|------|---------|---------------|
| LLM 特有邏輯 | Envoy 不懂 token、不懂 prompt | 深度理解 LLM 語意 |
| 路由決策 | 基於 HTTP 層（URL、header） | 基於模型、成本、token quota |
| 回應轉換 | 不轉換（透明代理） | 不同 Provider → 統一格式 |
| 串流處理 | 透明轉發 | 解析 SSE、計算 token、觸發 callback |
| 部署複雜度 | 需要 K8s + Service Mesh | 一個 Python 進程 |

**結論**：通用 Service Mesh 處理不了 LLM 的領域邏輯（token 計算、prompt 轉換、模型特有的錯誤碼）。LiteLLM 是**領域感知的代理**。

### 6. 雙層快取（DualCache）— 為什麼不只用 Redis？

#### LiteLLM 的做法

```python
class DualCache:
    def __init__(self):
        self.in_memory_cache = InMemoryCache()
        self.redis_cache = RedisCache()

    async def async_get_cache(self, key):
        # 1. 先查本地
        result = self.in_memory_cache.get(key)
        if result is not None:
            return result

        # 2. 本地沒有，查 Redis
        result = await self.redis_cache.async_get_cache(key)
        if result is not None:
            # 回填本地快取
            self.in_memory_cache.set(key, result)
        return result

    async def async_set_cache(self, key, value, ttl):
        # 同時寫入兩層
        self.in_memory_cache.set(key, value, ttl=ttl)
        await self.redis_cache.async_set_cache(key, value, ttl=ttl)
```

#### 替代方案比較

**方案 A：只用 Redis**
```python
# ❌
async def get_cooldown(deployment_id):
    return await redis.get(f"cooldown:{deployment_id}")
    # 每次路由決策都要一次網路往返（~1ms）
    # 高流量下：1000 QPS × 10 deployments = 10000 次/秒 Redis 查詢
```

**方案 B：只用 InMemory**
```python
# ❌
def get_cooldown(deployment_id):
    return memory_cache.get(f"cooldown:{deployment_id}")
    # 快，但多個 Gateway 節點之間不同步
    # Node 1 冷卻了某個 deployment，Node 2 不知道
```

**方案 C：DualCache（LiteLLM 的選擇）**
```python
# ✅
# 本地查詢：0.001ms（dict lookup）
# Redis 同步：最終一致，但不影響查詢延遲
```

| 考量 | 只 Redis | 只 InMemory | DualCache |
|------|----------|-------------|-----------|
| 延遲 | ~1ms（網路） | ~0.001ms | ~0.001ms |
| 一致性 | 強一致 | 無一致性 | 最終一致 |
| 跨節點同步 | ✅ | ❌ | ✅ |
| Redis 掛了 | 全部失敗 | N/A | 本地仍可用 |
| 適合場景 | 精確計費 | 單節點 | 冷卻狀態（容許短暫不一致） |

**關鍵洞察**：Cooldown Cache 不需要強一致——Node 1 冷卻了一個 deployment，Node 2 晚 1 秒知道也沒關係（頂多多送一個失敗請求）。但計費就需要強一致——多扣或少扣錢都是問題。

## 二、反模式：LiteLLM 刻意避開的設計

### 反模式 1：過度泛化

```python
# ❌ 有人可能想設計一個「萬用 AI 呼叫介面」
class UniversalAICall:
    def call(self, task_type, **kwargs):
        if task_type == "text":
            return self.completion(kwargs)
        elif task_type == "image":
            return self.image_gen(kwargs)
        elif task_type == "audio":
            return self.transcription(kwargs)
        elif task_type == "embedding":
            return self.embedding(kwargs)
```

**LiteLLM 的選擇**：保持 `completion()`、`embedding()`、`image_generation()` 為獨立函式。

**為什麼？**
- 不同 task 的參數完全不同（`messages` vs `input` vs `image_url`）
- 回傳格式不同
- 錯誤處理邏輯不同
- 強行統一 = 最低公約數，失去每個 task 的特有功能

### 反模式 2：Registry Pattern 做 Provider 管理

```python
# ❌ 有人可能這樣做
class ProviderRegistry:
    _registry = {}

    @classmethod
    def register(cls, name: str, config_class: type):
        cls._registry[name] = config_class

    @classmethod
    def get(cls, name: str):
        return cls._registry[name]()

# 每個 provider 檔案底部
ProviderRegistry.register("openai", OpenAIConfig)
ProviderRegistry.register("anthropic", AnthropicConfig)
```

**LiteLLM 怎麼做？** 直接用 model prefix mapping：

```python
# litellm 的做法：簡單的 prefix → config 映射
model_to_config = {
    "gpt-": OpenAIConfig,
    "claude-": AnthropicConfig,
    "gemini/": GeminiConfig,
    "azure/": AzureConfig,
}

def get_config_for_model(model: str):
    for prefix, config_cls in model_to_config.items():
        if model.startswith(prefix):
            return config_cls()
```

**為什麼不用 Registry？**

Registry 的問題是**隱式依賴**：
- 你不知道 `ProviderRegistry.get("openai")` 在哪裡被 register 的
- import 順序影響 registry 狀態（忘了 import → 找不到 provider）
- 測試時難以控制 registry 狀態

直接映射雖然「不夠 OOP」，但一眼就能看出有哪些 provider、怎麼匹配的。**可讀性 > 抽象性**。

### 反模式 3：Microkernel Architecture

```python
# ❌ Plugin 系統
class LiteLLMKernel:
    def __init__(self):
        self.plugins = []

    def load_plugin(self, plugin_path):
        module = importlib.import_module(plugin_path)
        self.plugins.append(module.Plugin())

    def completion(self, model, messages):
        for plugin in self.plugins:
            model, messages = plugin.pre_process(model, messages)
        result = self.core_completion(model, messages)
        for plugin in self.plugins:
            result = plugin.post_process(result)
        return result
```

**為什麼 LiteLLM 不做 plugin 系統？**

| 考量 | Plugin 系統 | LiteLLM 的 monolith |
|------|------------|---------------------|
| 新增 Provider | pip install litellm-anthropic | 內建，直接用 |
| 版本相容 | Plugin 和 core 版本對不上 = 爆 | 統一版本，CI 一起測 |
| 使用者體驗 | 要先找 plugin、安裝、設定 | `litellm.completion("claude-3-5-sonnet", ...)` 直接跑 |
| 維護成本 | N 個 plugin repo × M 個版本 | 一個 repo 統一維護 |

LiteLLM 選擇「大一統」而非 plugin——因為**對使用者最簡單**。trade-off 是 package size 較大，但在 AI 基礎設施的場景中這不是問題。

## 三、AI 工程核心概念

### Token 經濟學

理解 token 是 AI 工程的基礎：

```
1 token ≈ 4 個英文字元 ≈ 0.75 個英文單字
1 token ≈ 1-2 個中文字元

GPT-4o 定價（2024）：
- Input:  $2.50 / 1M tokens
- Output: $10.00 / 1M tokens
- Cached: $1.25 / 1M tokens（50% 折扣）

Claude Sonnet 4 定價：
- Input:  $3.00 / 1M tokens
- Output: $15.00 / 1M tokens
- Cache write: $3.75 / 1M tokens
- Cache read:  $0.30 / 1M tokens（90% 折扣！）
```

**核心洞察**：
- Output token 比 input token 貴 3-5 倍
- 快取是成本優化的最大槓桿
- 一次呼叫的成本 = f(模型選擇, input 長度, output 長度, 是否快取)

### Context Window 管理

```
┌────────────────────────────────────────┐
│           Context Window               │
│                                        │
│  ┌──────────┐ ┌──────────┐ ┌───────┐  │
│  │ System   │ │ History  │ │ Query │  │
│  │ Prompt   │ │ Messages │ │       │  │
│  │ (固定)   │ │ (動態)   │ │(變動) │  │
│  └──────────┘ └──────────┘ └───────┘  │
│                                        │
│  ← 越長越貴，但也越有上下文 →           │
└────────────────────────────────────────┘
```

**策略**：
1. **滑動視窗**：只保留最近 N 條歷史
2. **摘要壓縮**：將舊歷史壓縮為摘要
3. **RAG 增強**：只在需要時檢索相關上下文
4. **分層快取**：利用供應商的 prompt caching

### 負載均衡的藝術

AI 系統的負載均衡與傳統 Web 不同：

```
傳統 Web：
- 請求短且均勻（~100ms）
- 無狀態
- CPU/Memory bound

AI 系統：
- 請求長且不均勻（100ms ~ 120s）
- Token 限制而非請求限制
- GPU bound，排隊嚴重
- Rate limit 是硬性約束
```

**LiteLLM 的解法**：
- 追蹤 TPM/RPM 而非 QPS
- 冷卻機制處理暫時性故障
- 延遲感知路由避免慢節點
- 權重配置處理異質部署

### 錯誤處理的層次

```
┌─────────────────────────────────────┐
│ 層次 1：參數錯誤（可預防）           │
│ → 驗證 + drop_params               │
├─────────────────────────────────────┤
│ 層次 2：暫時性錯誤（可重試）         │
│ → 429 Rate Limit → 指數退避重試    │
│ → 500 Server Error → 換一個 deployment │
├─────────────────────────────────────┤
│ 層次 3：永久性錯誤（需 fallback）    │
│ → Context too long → 換更大的模型   │
│ → Content filter → 換其他供應商     │
├─────────────────────────────────────┤
│ 層次 4：系統級錯誤（需告警）         │
│ → 所有 deployment 都失敗            │
│ → 預算用完                         │
└─────────────────────────────────────┘
```

## 四、設計決策總結表

| 場景 | LiteLLM 的選擇 | 捨棄的替代方案 | 為什麼 |
|------|----------------|--------------|--------|
| 多 Provider 統一 | Adapter（Config 組合） | 繼承層次 | Config 和 Handler 可獨立變化 |
| 路由選擇 | Strategy（策略字串） | Chain of Responsibility | 路由是選擇問題，不是過濾問題 |
| 事後通知 | Observer（Callback list） | Event Bus | 單進程不需要 pub-sub 的重量級 |
| 可攔截檢查 | Pre-call Hook（Middleware） | Observer | 需要攔截請求、拋異常 |
| 橫切關注點 | Decorator（`@client`） | AOP 框架 | Python 原生語法夠用 |
| 快取 | DualCache（InMemory + Redis） | 只 Redis | 冷卻查詢不能有網路延遲 |
| Provider 發現 | Prefix mapping | Plugin Registry | 可讀性 > 抽象性，使用者零配置 |
| 系統架構 | Monolith SDK | Microkernel + Plugin | 使用者體驗：一行就能用 |
| 部署健康管理 | Per-deployment Cooldown | 全域 State Machine | 粒度更細，自動恢復 |
| Gateway 模式 | Application-level Proxy | Sidecar (Envoy) | 需要理解 LLM 領域語意 |

## 七、從源碼閱讀中獲得的洞察

### 1. 好的抽象比好的程式碼更重要

LiteLLM 的 `BaseConfig` 抽象讓 100+ 供應商的整合變得可管理。一個好的抽象可以讓系統在不修改核心的情況下擴展。

### 2. 全域狀態有時是務實的選擇

`litellm.success_callback = [...]` 這種全域狀態設計被許多人批評，但它確實讓使用者的程式碼極其簡潔。工程是取捨的藝術。

### 3. 不是每個地方都需要最「正確」的 pattern

`simple_shuffle` 路由策略就是一個 `random.choice()`。看起來「太簡單了吧」？但對 80% 的使用者來說，這就是最好的選擇——零配置、零認知負擔、效果夠好。

### 4. 可觀測性不是事後的想法

從一開始就嵌入 logging、cost tracking、callbacks——這些不是「nice to have」，是生產環境的必需品。

### 5. 選擇無聊的技術

LiteLLM 用 dict 不用 Redis（在 InMemory 層）、用 list 不用 Event Bus、用 prefix matching 不用 Plugin Registry。每個「無聊」的選擇都減少了一個可能出錯的地方。

## 給初學者的最後建議

1. **不要因為知道設計模式就強行使用它們**。先寫最簡單能用的版本，等痛點出現了再重構。
2. **讀源碼時問「為什麼不用 X」比「這是什麼 pattern」更有價值**。理解被捨棄的選項才是真正理解設計決策。
3. **AI 領域變化極快**。今天學的具體 API 明天可能就變了，但設計模式和系統思維是持久的。
4. **建立自己的筆記**。把「我最初不理解但後來搞懂」的東西記下來——這就是最好的教學材料。
5. **行動是最好的學習**。選一個具體目標去實作，比讀十本書有用。
