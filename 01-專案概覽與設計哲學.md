## LiteLLM 是什麼？

LiteLLM 是一個**統一的 LLM API 介面層**，它讓開發者用同一套程式碼呼叫超過 100 個不同的 AI 模型供應商。

**白話版**：想像你有 100 種不同插座（美規、歐規、英規...），LiteLLM 就是那個「萬用轉接頭」——你只需要一種插頭，它幫你接上任何插座。

### 核心問題：API 碎片化

在 AI 工程的實際工作中，你會面對這樣的困境：

```python
# ❌ 沒有 LiteLLM 的世界 — 每個供應商都有不同的 API
import openai
import anthropic
import google.generativeai

# OpenAI 的呼叫方式
response = openai.ChatCompletion.create(
    model="gpt-4",
    messages=[{"role": "user", "content": "Hello"}]
)

# Anthropic 的呼叫方式（完全不同的格式）
response = anthropic.messages.create(
    model="claude-3-opus-20240229",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Hello"}]
)

# Google 又是另一種格式...
```

```python
# ✅ 有 LiteLLM 的世界 — 統一介面
import litellm

# 同一個函式，只需改 model 參數
response = litellm.completion(
    model="gpt-4",
    messages=[{"role": "user", "content": "Hello"}]
)

response = litellm.completion(
    model="claude-3-opus-20240229",
    messages=[{"role": "user", "content": "Hello"}]
)

response = litellm.completion(
    model="gemini/gemini-pro",
    messages=[{"role": "user", "content": "Hello"}]
)
```

## 設計哲學

### 1. OpenAI 格式作為通用語言

LiteLLM 選擇了 OpenAI 的 API 格式作為統一標準。這不是因為 OpenAI 最好，而是因為：
- 它是業界最廣泛採用的格式
- 大多數開發者已經熟悉
- 大多數工具和框架都支援

> **給初學者的類比**：就像全世界的程式設計師都用英文寫 commit message 一樣——不是英文最好，而是它是大家都懂的「共通語言」。OpenAI 的 API 格式在 AI 領域扮演了相同角色。

### 2. 兩層架構

```
┌─────────────────────────────────────────────┐
│           LiteLLM Gateway (Proxy)           │  ← 企業級功能
│  認證 | 限流 | 預算 | 負載均衡 | 日誌      │
├─────────────────────────────────────────────┤
│              LiteLLM SDK                    │  ← 核心轉換
│  統一介面 | 格式轉換 | 串流 | 成本計算     │
├─────────────────────────────────────────────┤
│    OpenAI | Anthropic | Google | Azure ...  │  ← 供應商 API
└─────────────────────────────────────────────┘
```

- **SDK 層**（`litellm/` 目錄）：負責 API 格式轉換、串流處理、錯誤映射
- **Gateway 層**（`litellm/proxy/` 目錄）：在 SDK 之上加入認證、限流、可觀測性

### 3. 適配器模式（Adapter Pattern）

每個供應商都有一個「轉換器」（Transformation Config），負責：
- 將 OpenAI 格式的請求 → 轉換為該供應商的原生格式
- 將供應商的原生回應 → 轉換為 OpenAI 格式的回應

```python
# 每個供應商實作 BaseConfig 的抽象方法
class AnthropicChatConfig(BaseConfig):
    def transform_request(self, ...):
        """把 OpenAI 格式轉為 Anthropic 格式"""
        
    def transform_response(self, ...):
        """把 Anthropic 回應轉為 OpenAI 格式"""
```

### 4. 最小侵入性

LiteLLM 的設計原則是**不要求你改變現有程式碼結構**：
- 你可以一行程式碼替換 `openai.ChatCompletion.create`
- 回應格式與 OpenAI SDK 完全相容
- 支援同步和非同步兩種模式

## 專案目錄結構

```
litellm/
├── __init__.py          # 全域變數、callback 設定
├── main.py              # 核心入口：completion(), acompletion(), embedding()
├── router.py            # 路由引擎：負載均衡、故障轉移
├── cost_calculator.py   # 成本計算引擎
├── utils.py             # 工具函式庫
├── constants.py         # 常數定義
├── exceptions.py        # 統一錯誤類型
│
├── llms/                # 所有供應商的適配器
│   ├── base_llm/        # 抽象基類 (BaseConfig)
│   ├── openai/          # OpenAI 實作
│   ├── anthropic/       # Anthropic 實作
│   ├── bedrock/         # AWS Bedrock 實作
│   ├── vertex_ai/       # Google Vertex AI 實作
│   ├── custom_httpx/    # 統一 HTTP 通訊層
│   └── ...              # 100+ 其他供應商
│
├── proxy/               # Gateway 伺服器
│   ├── proxy_server.py  # FastAPI 路由端點
│   ├── auth/            # 認證模組
│   └── hooks/           # 前置/後置掛勾
│
├── caching/             # 快取系統
├── integrations/        # 第三方可觀測性整合
├── router_strategy/     # 路由策略演算法
└── types/               # 型別定義
```

## 關鍵數據

- **支援 100+ LLM 供應商**（截至 2026 年）
- **GitHub Stars**: 15k+
- **核心程式碼**: Python，使用 FastAPI + httpx
- **主要依賴**: openai, httpx, pydantic, tiktoken

## 動手試試看

如果你已經安裝了 litellm（`pip install litellm`），試跑這段程式碼：

```python
import litellm
import os

# 只需要一個 API Key 就能開始（任選一個你有的）
# os.environ["OPENAI_API_KEY"] = "sk-..."
# os.environ["ANTHROPIC_API_KEY"] = "sk-ant-..."

# 列出 LiteLLM 支援的所有模型（不需要 API Key）
print(f"LiteLLM 版本: {litellm.__version__}")
print(f"支援的供應商數量: {len(litellm.provider_list)}")
print(f"前 10 個供應商: {litellm.provider_list[:10]}")
```

> **常見問題**：「我沒有任何 API Key 怎麼辦？」—— 沒關係！本教學的重點是**理解源碼設計**，不是實際呼叫 API。你可以純粹閱讀程式碼來學習。如果想動手實驗，Gemini 和 DeepSeek 提供免費額度。
