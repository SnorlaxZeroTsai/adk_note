## 21. The Quiet Contributions: Silent AI PRs

**論文標題：** The Quiet Contributions: Characterizing the Impact of Silent AI-Generated Pull Requests on Code Complexity and Security  
**arXiv ID：** 2601.21102

**中文摘要：**  
本研究探討「靜默」AI 生成的 Pull Request（即未明確標示為 AI 產出的 PR）對程式碼複雜度與安全性的影響。研究分析大量開源專案中可能由 AI 悄悄貢獻的程式碼變更，評估其對圈複雜度、認知複雜度及潛在安全漏洞的影響。結果顯示，靜默 AI PR 在某些情境下會增加程式碼複雜度且引入安全隱患。

**核心貢獻：**
- 提出識別「靜默」AI PR 的方法論框架
- 量化 AI 生成程式碼對複雜度指標（圈複雜度、認知複雜度）的影響
- 分析 AI 生成程式碼中常見的安全漏洞模式
- 揭示未標記 AI 貢獻對開源生態系統的潛在風險

**方法論：**  
透過行為特徵分析與統計方法識別可能的靜默 AI PR，使用靜態分析工具測量程式碼複雜度變化，並以安全掃描工具偵測潛在漏洞，對比人工 PR 與 AI PR 的差異。

**主要發現：**
- 靜默 AI PR 佔開源貢獻的比例正在快速增加
- AI 生成的程式碼在某些模式下傾向增加不必要的複雜度
- 安全相關的程式碼變更中，AI PR 更容易引入特定類型的漏洞
- 程式碼審查流程對靜默 AI PR 的偵測能力有限

**研究限制：**
- 識別「靜默」AI PR 的方法依賴行為特徵推斷，存在假陽性與假陰性風險，無法確認所有被標記的 PR 確實為 AI 生成
- 研究僅涵蓋開源專案，企業內部的靜默 AI 使用情況可能截然不同且無法透過此方法研究
- 靜態分析工具對安全漏洞的偵測覆蓋率有限，可能遺漏更深層的邏輯漏洞或特定語言的安全問題

**其他資訊：**
- 與論文 #33（Fingerprinting AI Coding Agents）的研究互補，兩者皆關注如何在未標記的情況下識別 AI 生成內容，可結合行為指紋技術提升偵測準確率
- 對開源專案維護者具有直接實務意義：需考慮建立 AI 生成內容揭露政策，並強化針對 AI 特有安全模式的審查流程

## 22. Agent Benchmarks Fail Public Sector

**論文標題：** Agent Benchmarks Fail Public Sector: A Gap Analysis of 1300+ Benchmarks vs Public-Sector Needs  
**arXiv ID：** 2601.20617

**中文摘要：**  
本研究對超過 1,300 個現有 AI Agent 基準測試進行系統性分析，揭示其與公共部門實際需求之間的巨大差距。研究發現現行基準測試過度集中於商業與學術場景，忽略了政府服務、公共安全、醫療保健等公共部門的獨特需求，例如合規性、公平性、多語言支持等。

**核心貢獻：**
- 建立 1,300+ 個 Agent 基準測試的全面分類目錄
- 識別公共部門的獨特需求維度（合規、公平、透明度等）
- 量化現有基準與公共部門需求間的覆蓋差距
- 提出公共部門專用基準測試的設計框架

**方法論：**  
對現有基準測試進行系統性文獻綜述與分類，同時透過訪談與需求分析確定公共部門的關鍵評估維度，最後進行差距分析比較兩者的覆蓋範圍。

**主要發現：**
- 現有基準測試極少涵蓋公共部門的核心需求
- 合規性驗證、公民公平性、跨機構協作等面向嚴重缺乏評估工具
- 公共部門需要考量的倫理與法規約束在現行基準中幾乎不存在
- 建議開發專門針對公共部門的多維度評估框架

**研究限制：**
- 公共部門需求的定義可能因國家、地區與政治體制不同而有顯著差異，研究結果的跨國適用性有限
- 1,300+ 基準測試的分類依賴研究者的判斷標準，不同分類方式可能產生不同的差距分析結論
- 研究指出差距但未提供經過驗證的解決方案，所提出的設計框架仍屬概念性質，尚未經過實證測試

**其他資訊：**
- 對政府 AI 採購決策者極具參考價值，可作為制定 AI Agent 驗收標準的起點
- 研究揭示的差距為學術界與產業界提供了明確的基準測試開發方向，尤其在多語言、公平性與合規驗證面向存在大量未開發空間

## 23. Interpreting Emergent Extreme Events

**論文標題：** Interpreting Emergent Extreme Events in Multi-Agent Systems Using Shapley Values  
**arXiv ID：** 2601.20538

**中文摘要：**  
本研究提出使用 Shapley 值方法來解釋多代理系統中突發極端事件的歸因問題。當多個 AI Agent 協同工作時，系統可能產生非預期的極端行為，本文提供了一個數學框架來量化每個 Agent 對極端事件的貢獻度，從而實現可解釋的責任歸屬。

**核心貢獻：**
- 將博弈論中的 Shapley 值引入多 Agent 系統的事件歸因
- 建立極端事件中各 Agent 貢獻度的量化框架
- 提供可解釋性方法以追蹤突發行為的來源
- 驗證該方法在不同規模的多 Agent 系統中的適用性

**方法論：**  
基於合作博弈論的 Shapley 值概念，將多 Agent 系統的輸出視為聯盟博弈的結果，計算每個 Agent 的邊際貢獻。透過模擬不同 Agent 組合的行為來估算各 Agent 對極端事件的貢獻值。

**主要發現：**
- Shapley 值能有效區分在極端事件中起主導作用的 Agent
- 該方法可識別出導致系統性風險的 Agent 交互模式
- 隨著 Agent 數量增加，需要近似算法來降低計算複雜度
- 歸因結果有助於改進系統設計以防止未來的極端事件

**研究限制：**
- Shapley 值的精確計算為 NP-hard 問題，隨 Agent 數量增加呈指數級膨脹，在大規模系統（數十個以上 Agent）中僅能使用近似方法，歸因精確度受限
- 假設各 Agent 的貢獻具有可加性與可分離性，但實際多 Agent 系統中的湧現行為可能源於高度非線性的交互作用，無法被線性歸因框架完全捕捉
- 模擬不同 Agent 組合需要大量運算資源，且假設移除某 Agent 不會改變其餘 Agent 的行為，此假設在強耦合系統中未必成立

**其他資訊：**
- 與論文 #27（Architecture-Aware Evaluation Metrics）的方法論可互補：Shapley 值提供 Agent 級歸因，架構感知指標提供組件級歸因，結合使用可建構更完整的系統行為解釋
- 在金融系統風險管理、多 Agent 自駕車協調等安全關鍵應用中具有實際部署潛力，可作為事後審計工具使用

## 24. Who Writes the Docs? Agent vs Human PRs

**論文標題：** Who Writes the Docs? Comparing Agent and Human Documentation Contributions in Pull Requests  
**arXiv ID：** 2601.20171

**中文摘要：**  
本研究比較 AI Agent 與人類開發者在 Pull Request 中的文件貢獻差異。分析涵蓋 README、API 文件、程式碼註解等不同類型的文件，評估 AI Agent 在文件撰寫方面的品質、完整性及與程式碼變更的一致性。結果揭示 Agent 文件貢獻的獨特模式與品質特徵。

**核心貢獻：**
- 系統性比較 AI Agent 與人類的文件貢獻模式
- 建立文件品質的多維度評估指標
- 識別 AI 生成文件的典型特徵與不足之處
- 提供改善 AI 文件生成的具體建議

**方法論：**  
收集包含文件變更的 PR 資料集，根據貢獻來源（AI Agent vs 人類）分組，使用自然語言處理工具評估文件品質（可讀性、準確性、完整性），並由人工評審進行補充驗證。

**主要發現：**
- AI Agent 傾向生成格式標準化但內容淺薄的文件
- 人類文件更常包含設計決策的脈絡說明
- Agent 在更新現有文件時常遺漏重要的上下文資訊
- AI 生成的 API 文件在結構完整性上表現較好，但在使用範例上表現較弱

**研究限制：**
- 文件「品質」的定義帶有主觀性，人工評審的判斷可能受到個人偏好與背景知識影響，不同領域對好文件的標準也不同
- 研究主要分析英文文件，在非英語文件或多語言專案中的結果可能不同
- NLP 工具對程式碼文件的可讀性評估可能不如對一般文本的評估準確，技術文件有其特殊語法結構

**其他資訊：**
- 與論文 #30（Making Every PR Meaningful）的發現一致：AI 在結構化、格式化任務表現較好，但在需要深度理解的任務上仍有差距
- 對 Agent 開發者有實務意義：改善 Agent 文件生成應著重於注入專案上下文與設計決策歷史，而非僅追求格式正確性

## 25. Are We All Using Agents the Same Way?

**論文標題：** Are We All Using Agents the Same Way? Differences Between Core and Peripheral Developers  
**arXiv ID：** 2601.20106

**中文摘要：**  
本研究調查核心開發者與外圍開發者在使用 AI Agent 方面的差異。研究發現兩類開發者在 Agent 使用頻率、任務類型、信任程度及程式碼審查行為上存在顯著差異。核心開發者更傾向將 Agent 用於複雜重構任務，而外圍開發者則偏好將 Agent 用於入門性質的貢獻。

**核心貢獻：**
- 揭示核心與外圍開發者使用 AI Agent 的行為差異
- 分析不同角色的開發者對 Agent 輸出的信任層級
- 研究 Agent 使用模式對開源社群參與動態的影響
- 提供提升不同開發者群體 Agent 體驗的建議

**方法論：**  
透過對開源專案的貢獻歷史分析，將開發者分為核心（高頻、長期貢獻者）與外圍（偶爾貢獻者）兩類，分析其 Agent 使用行為、PR 接受率及互動模式的差異。

**主要發現：**
- 核心開發者使用 Agent 處理更複雜的任務，但更頻繁修改 Agent 的輸出
- 外圍開發者更傾向直接提交 Agent 生成的程式碼而不做修改
- Agent 使用降低了外圍開發者的參與門檻
- 核心開發者對 Agent 生成程式碼的審查標準更為嚴格

**研究限制：**
- 核心與外圍開發者的分類標準（貢獻頻率）可能過於簡化，未考慮專業知識深度、角色職責等其他重要維度
- 研究基於觀察性資料，難以建立因果關係——究竟是角色決定了 Agent 使用方式，還是使用方式影響了開發者角色的演變
- 樣本可能偏向已採用 Agent 的活躍專案，不代表整體開源生態系統的情況

**其他資訊：**
- 對開源社群治理具有重要啟示：外圍開發者直接提交未修改的 Agent 程式碼可能增加維護負擔，需考慮建立適當的貢獻指南
- 與論文 #36（Will It Survive）結合閱讀可探討：不同開發者類型提交的 AI 程式碼是否有不同的存活率模式

## 26. DevOps-Gym: AI Agents in DevOps Cycle

**論文標題：** DevOps-Gym: Benchmarking AI Agents in the DevOps Cycle with 700+ Real-World Tasks  
**arXiv ID：** 2601.20882

**中文摘要：**  
本研究提出 DevOps-Gym，一個包含 700 多個真實世界 DevOps 任務的基準測試平台。該平台涵蓋持續整合、部署、監控、事故回應等 DevOps 生命週期的各個階段，用以全面評估 AI Agent 在 DevOps 場景中的能力與侷限。

**核心貢獻：**
- 建立包含 700+ 真實 DevOps 任務的綜合基準測試
- 覆蓋 DevOps 生命週期的多個關鍵階段
- 提供標準化的評估框架與可重現的測試環境
- 對多個主流 AI Agent 進行系統性能力評估

**方法論：**  
從真實 DevOps 場景中收集任務，分類標註難度等級與所需技能，在沙箱環境中部署可重現的測試基礎設施，使用自動化評估指標衡量 Agent 的任務完成度與效率。

**主要發現：**
- 現有 AI Agent 在簡單配置任務上表現良好，但在複雜故障排除場景中表現不佳
- Agent 在需要跨多個工具鏈協作的任務上困難明顯
- 監控與告警配置類任務的完成率相對較高
- 安全相關的 DevOps 任務是 Agent 的主要弱項之一

**研究限制：**
- 沙箱環境無法完全模擬真實 DevOps 場景的複雜性，如網路延遲、資源競爭、多團隊並行操作等因素被簡化
- 700+ 任務雖已有相當規模，但 DevOps 領域極為廣泛，某些特定工具鏈或雲端平台的覆蓋可能不足
- 基準測試可能快速過時，因為 DevOps 工具與最佳實踐持續演進，評估結果具有時效性

**其他資訊：**
- 與論文 #32（AI Agents Touch CI/CD Configurations）形成互補，DevOps-Gym 提供量化基準，論文 #32 提供質性行為分析
- 對企業 DevOps 團隊評估 AI Agent 導入可行性具有直接參考價值，可根據任務類型判斷哪些環節適合引入 Agent 輔助

## 27. Architecture-Aware Evaluation Metrics

**論文標題：** Architecture-Aware Evaluation Metrics: Linking Components to Behaviors  
**arXiv ID：** 2601.19583

**中文摘要：**  
本研究提出架構感知的評估指標，將 AI Agent 系統的組件特性與其最終行為表現建立關聯。傳統評估僅關注端到端表現，無法解釋哪個組件對特定行為負責。本文建立的方法論能追蹤從架構組件到可觀察行為的因果路徑。

**核心貢獻：**
- 提出將系統架構與行為表現連結的評估指標
- 建立組件層級的貢獻歸因方法
- 設計可分解的評估框架以識別效能瓶頸
- 驗證該方法在多種 Agent 架構上的適用性

**方法論：**  
定義架構組件（如規劃器、記憶模組、工具呼叫器等）的功能邊界，透過消融實驗與因果推斷方法，量化每個組件對最終行為指標的貢獻度，建立組件-行為的對映關係。

**主要發現：**
- 架構感知指標能有效識別系統中的效能瓶頸組件
- 不同任務類型對組件的依賴模式差異顯著
- 規劃模組在複雜任務中的重要性遠超記憶模組
- 組件間的交互效果佔整體行為變異的顯著比例

**研究限制：**
- 消融實驗假設組件可被獨立移除或替換，但高度耦合的系統中組件互依性可能使消融結果難以解讀
- 方法需要明確的架構組件邊界定義，對於端到端訓練或組件邊界模糊的系統適用性降低
- 因果推斷方法的有效性依賴於充足的實驗樣本，計算成本可能在大型系統中變得不可接受

**其他資訊：**
- 與論文 #23（Shapley Values for Extreme Events）方法論互補，兩者皆處理歸因問題但在不同粒度層級運作
- 對 Agent 系統的開發者具有直接工程價值：可用於指導架構優化決策，優先改進對效能影響最大的組件

## 28. Balancing Sustainability And Performance

**論文標題：** Balancing Sustainability and Performance: Smaller LLMs for Energy Reduction  
**arXiv ID：** 2601.19311

**中文摘要：**  
本研究探討在 AI Agent 應用中使用較小型 LLM 以實現能源節約與永續發展的可能性。透過系統性比較不同規模模型在各類任務上的效能與能耗，研究找出在維持可接受效能的前提下能顯著降低能源消耗的模型配置策略。

**核心貢獻：**
- 量化不同規模 LLM 的能源消耗與效能權衡
- 提出任務感知的模型選擇策略以優化能效比
- 建立永續性評估的多維度指標體系
- 驗證小型模型在特定任務上可達到足夠的效能水準

**方法論：**  
對多種規模的 LLM（從數十億到數千億參數）在標準化任務套件上進行效能與能耗測量，使用帕累托前沿分析識別最佳效能-能耗平衡點，並根據任務複雜度提出動態模型選擇策略。

**主要發現：**
- 較小模型在簡單任務上可達到大模型 90% 以上的效能，而能耗僅為其 10-20%
- 任務複雜度是決定最佳模型規模的關鍵因素
- 動態路由策略可在幾乎不損失效能的情況下減少 40-60% 的能源消耗
- 永續性考量應成為 Agent 系統設計的核心指標之一

**研究限制：**
- 能源消耗的測量方式可能因硬體環境、批次大小、推論優化等因素而大幅變動，研究結果難以在不同部署環境中直接對比
- 動態路由策略的「任務複雜度」判斷本身需要額外計算，此路由開銷未必在所有情境下都能被節省的能耗覆蓋
- 模型能力的快速演進使研究結果具有時效性，較小模型的能力邊界持續提升可能改變最佳配置

**其他資訊：**
- 隨著 ESG 法規日趨嚴格，此研究為企業提供了量化 AI 系統碳足跡與選擇能效最佳配置的方法論基礎
- 動態路由概念可與 MoE（Mixture of Experts）架構研究結合，在模型內部與模型間兩個層級同時進行效率優化

## 29. Dominant Themes in Reviewing Agent Code

**論文標題：** Dominant Themes in Reviewing Agent-Generated Code: A Taxonomy of 12 Review Themes  
**arXiv ID：** 2601.19287

**中文摘要：**

## 36. Will It Survive? Fate of AI-Generated Code

**論文標題：** Will It Survive? A Survival Analysis of the Fate of 200K+ AI-Generated Code Units  
**arXiv ID：** 2601.16809

**中文摘要：**  
本研究對超過 200,000 個 AI 生成的程式碼單元進行存活分析，追蹤其在專案中的生命週期——從合併到被修改或刪除。研究揭示 AI 程式碼的存活曲線特徵，以及影響程式碼存活的關鍵因素，如程式碼類型、專案活躍度、Agent 來源等。

**核心貢獻：**
- 對 200K+ AI 生成程式碼單元進行縱向存活分析
- 建立 AI 程式碼生命週期的統計模型
- 識別影響 AI 程式碼存活率的關鍵預測因子
- 比較 AI 與人類程式碼的存活曲線差異

**方法論：**  
使用存活分析統計方法（如 Kaplan-Meier 估計、Cox 比例風險模型），追蹤 AI 生成程式碼從合併到首次重大修改或刪除的時間，分析各種協變量（程式碼類型、規模、Agent 來源等）對存活率的影響。

**主要發現：**
- AI 生成的程式碼中位存活時間顯著短於人類程式碼
- 測試程式碼的存活率高於生產程式碼
- 大型 AI 生成的程式碼單元更容易在短期內被修改
- 經過嚴格審查的 AI 程式碼存活時間顯著延長

**研究限制：**
- 存活分析中「修改」事件的定義可能過於寬泛，小幅格式調整與重大邏輯重寫被同等計入，可能低估 AI 程式碼的實質穩定性
- 右截斷（right-censoring）問題——較新的程式碼尚未經歷足夠長的觀察期，可能影響存活曲線估計的準確性
- 無法完全排除混淆因素：AI 程式碼可能傾向被引入變動較快的模組中，而非其本身品質較差導致短存活

**其他資訊：**
- 200K+ 的資料規模使統計分析結果具有較高可信度，Cox 比例風險模型可提供具有實用價值的預測因子
- 「經過嚴格審查的 AI 程式碼存活時間顯著延長」這一發現直接支持加強程式碼審查投入的決策，具有明確的 ROI 論證價值

## 37. LUMINA: Long-horizon Multi-turn Agents

**論文標題：** LUMINA: A Long-horizon Multi-turn Agent Evaluation Framework with Oracle Counterfactual  
**arXiv ID：** 2601.16649

**中文摘要：**  
本研究提出 LUMINA 框架，用於評估長期多輪互動的 AI Agent。核心創新在於引入「Oracle 反事實」機制——透過比較 Agent 的實際決策與全知最優決策之間的差距，更精確地評估 Agent 在長期任務中每個轉折點的決策品質。

**核心貢獻：**
- 提出針對長期多輪 Agent 的評估框架 LUMINA
- 引入 Oracle 反事實方法以量化決策品質
- 解決長序列評估中的信用分配問題
- 提供比端到端指標更細粒度的診斷能力

**方法論：**  
建構具有已知最優解的任務環境，在每個決策點記錄 Agent 的選擇並與 Oracle（全知）策略進行比較，計算反事實偏差分數，並分析錯誤在多輪互動中的累積效應。

- 早期錯誤具有放大效應，對最終結果影響不成比例
- LUMINA 能有效區分不同能力層級的 Agent

**研究限制：**
- Oracle 反事實方法需要已知最優解的任務環境，在開放式或無唯一最優解的真實任務中難以應用
- 「全知最優策略」的建構本身可能不完美，若 Oracle 有誤則所有基於反事實偏差的評估都會失準
- 框架的計算成本隨任務長度與決策空間增大而快速增長，在超長序列任務中可能不實用

**其他資訊：**
- 與論文 #39（Agentic Confidence Calibration）在方法論上互補：LUMINA 從外部評估決策品質，信心校準從內部評估 Agent 的自我認知
- 信用分配問題是強化學習的核心挑戰之一，LUMINA 的 Oracle 反事實方法為此提供了一個新穎的監督信號來源

## 38. Tool Invocation Reliability

**論文標題：** Tool Invocation Reliability: A 12-Category Error Taxonomy  
**arXiv ID：** 2601.16280

**中文摘要：**  
本研究建立了 AI Agent 工具呼叫錯誤的 12 類分類法。透過大規模分析 Agent 在實際任務中的工具呼叫失敗案例，系統性歸納出常見的錯誤模式，包括參數格式錯誤、工具選擇不當、回應解析失敗等類別，為提升工具呼叫可靠性提供診斷框架。

**核心貢獻：**
- 建立包含 12 個類別的工具呼叫錯誤分類法
- 量化各類錯誤的發生頻率與影響程度
- 分析錯誤發生的上下文條件與觸發因素
- 提出針對各類錯誤的緩解策略

**方法論：**  
收集多個 Agent 系統的工具呼叫日誌，標記失敗案例並進行根因分析，透過迭代編碼方法歸納出穩定的錯誤分類，最後量化各類別的分佈與嚴重程度。

**主要發現：**
- 參數格式不匹配是最常見的工具呼叫錯誤類別
- 約 30% 的錯誤源於工具選擇不當（選錯工具）
- 複雜工具鏈中的錯誤傳播是可靠性下降的主要原因
- 適當的錯誤恢復機制可將整體失敗率降低 40% 以上

**研究限制：**
- 錯誤分類法基於當前 Agent 與工具的互動模式，隨著工具介面標準化（如 MCP 協議）的推進，某些錯誤類別可能自然消失
- 不同 Agent 框架的工具呼叫機制差異可能影響錯誤分佈，研究結果的跨框架泛化性需要進一步驗證
- 僅關注可觀察的失敗案例，未涵蓋「靜默錯誤」（即工具呼叫成功但結果不正確）的情況

**其他資訊：**
- 12 類錯誤分類法可直接用於建構工具呼叫的自動化測試套件，與論文 #31（Automated Structural Testing）的方法結合應用
- 對 Agent 框架設計者有實務價值：根據錯誤頻率優先改善參數驗證與工具選擇邏輯，可最大化可靠性提升效果

## 39. Agentic Confidence Calibration

**論文標題：** Agentic Confidence Calibration: Holistic Trajectory Calibration  
**arXiv ID：** 2601.15778

**中文摘要：**  
本研究提出「整體軌跡校準」（Holistic Trajectory Calibration）方法，用於校準 AI Agent 在多步驟任務中的信心度。不同於單步校準，該方法考慮整條決策軌跡的信心一致性，使 Agent 能更準確地評估其在完成整個任務過程中的確定性程度。

**核心貢獻：**
- 提出針對 Agent 決策軌跡的整體信心校準方法
- 解決多步驟任務中信心度估計的累積偏差問題
- 建立軌跡層級的校準誤差指標
- 驗證方法在多種 Agent 架構上的有效性

**方法論：**  
定義軌跡級信心分數的計算方式，設計校準目標函數以最小化軌跡級校準誤差（ECE），提出基於後處理的校準方法與基於訓練的校準方法，在多個 Agent 基準上進行評估。

**主要發現：**
- 傳統步驟級校準在多步驟任務中嚴重低估不確定性
- 整體軌跡校準能顯著改善 Agent 的「知其所不知」能力
- 良好校準的信心度可用於觸發人類介入或回退機制
- 長期任務中的校準退化問題需要動態調整策略

**研究限制：**
- 校準方法依賴大量帶有真實標籤的軌跡資料進行訓練，在資料稀缺的新領域中冷啟動問題嚴重
- 軌跡級校準假設任務有明確的成功/失敗判定標準，對於結果具有連續性或多維度評價的任務適用性較低
- 動態環境中任務難度隨時間變化，靜態校準模型可能無法適應分佈漂移

**其他資訊：**
- 在人機協作系統中具有高度實用價值：校準良好的信心度可作為「何時請求人類幫助」的決策依據，提升人機團隊效率
- 與論文 #37（LUMINA）互補：LUMINA 提供外部決策品質評估，信心校準提供內部不確定性估計，兩者結合可建構更完整的 Agent 自我監控系統

## 40. Improving Agentic Evaluations Across Domains

**論文標題：** Improving Agentic Evaluations Across Domains: Methodological Challenges and Best Practices  
**arXiv ID：** 2601.15679

**中文摘要：**  
本研究系統性地探討跨領域 Agent 評估中的方法論挑戰與最佳實踐。研究識別出評估設計、指標選擇、環境設定、結果報告等方面的常見陷阱，並提出一套改善 Agent 評估品質與可比較性的方法論建議，旨在提升整個領域的評估標準。

**核心貢獻：**
- 全面盤點跨領域 Agent 評估的方法論挑戰
- 識別評估設計中的常見偏差與陷阱
- 提出改善評估可重現性與可比較性的最佳實踐
- 建立 Agent 評估品質的自我檢查清單

**方法論：**  
對現有 Agent 評估研究進行後設分析（meta-analysis），從評估設計、資料收集、指標定義、結果分析等維度系統性識別方法論問題，並綜合最佳實踐形成建議框架。

**主要發現：**
- 多數 Agent 評估缺乏足夠的環境控制，導致結果難以重現
- 單一指標評估會嚴重低估 Agent 的能力邊界
- 人類基線的缺失是跨領域比較的最大障礙
- 評估任務的飽和度問題（ceiling effects）日益嚴重，需要持續更新基準

**研究限制：**
- 作為後設分析研究，其結論的品質受限於所分析的原始研究之品質與多樣性
- 所提出的最佳實踐建議較為通用，不同領域（如醫療、金融、軟體工程）的具體實施方式可能需要大幅調整
- 評估方法論本身也在快速演進中，當前的最佳實踐可能在 Agent 能力跨越新門檻後需要根本性修訂

**其他資訊：**
- 該研究可作為 Agent 評估領域的方法論指南，對研究者設計新基準測試具有直接參考價值
- 與本集合中多篇論文（#22 公共部門差距、#26 DevOps-Gym、#37 LUMINA）的評估設計呼應，為這些具體基準提供了方法論品質提升的建議框架
