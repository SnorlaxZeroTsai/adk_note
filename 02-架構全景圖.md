> **本章目標**：建立 LiteLLM 整體架構的心智模型。讀完後你看到任何 LiteLLM 的程式碼，都能知道它屬於哪一層、扮演什麼角色。
>
> **預計閱讀時間**：20 分鐘
>
> **前置閱讀**：[第一章：專案概覽](./01-專案概覽與設計哲學.md)

## 請求的完整生命週期

當你呼叫 `litellm.completion()` 時，背後發生了什麼？讓我們追蹤一個請求從頭到尾的旅程。

> **給初學者的比喻**：想像你在一家高級餐廳點餐。你告訴服務生「我要一份牛排」（統一介面），服務生把你的需求翻譯給廚房（格式轉換），廚房做好後服務生再端回來給你（回應轉換）。LiteLLM 就是那個服務生——你不需要知道廚房用的是什麼設備、什麼調理方式。

### SDK 模式（直接呼叫）

```
使用者程式碼
    │
    ▼
litellm.completion(model="claude-3-opus", messages=[...])
    │
    ▼
┌─────────────────────────────────────────┐
│  main.py :: completion()                 │
│  1. 解析 model 字串，判斷供應商           │
│  2. 取得對應的 ProviderConfig             │
│  3. 設定 logging 物件                    │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│  BaseLLMHTTPHandler :: completion()       │
│  1. 呼叫 config.transform_request()      │
│  2. 呼叫 config.get_complete_url()       │
│  3. 發送 HTTP 請求                       │
│  4. 呼叫 config.transform_response()     │
└────────────────┬────────────────────────┘
                 │
                 ▼
┌─────────────────────────────────────────┐
│  回傳 ModelResponse（OpenAI 格式）        │
└─────────────────────────────────────────┘
```

## 五大核心子系統

### 1. 轉換子系統（Transformation）

**位置**: `litellm/llms/*/chat/transformation.py`

負責將 OpenAI 格式的請求/回應與各供應商的原生格式互相轉換。

```python
# 概念示意
class AnthropicConfig(BaseConfig):
    def transform_request(self, model, messages, optional_params, ...):
        # OpenAI messages → Anthropic messages 格式
        return {
            "model": model,
            "messages": self._transform_messages(messages),
            "max_tokens": optional_params.get("max_tokens", 4096),
        }
    
    def transform_response(self, raw_response, ...):
        # Anthropic 回應 → OpenAI ModelResponse 格式
        return ModelResponse(
            choices=[Choices(message=Message(...))],
            usage=Usage(...)
        )
```

### 2. 通訊子系統（HTTP Handler）

**位置**: `litellm/llms/custom_httpx/llm_http_handler.py`

統一管理所有與供應商的 HTTP 通訊，包含：
- 連接池管理
- 超時控制
- SSL 處理
- 串流回應處理

### 3. 路由子系統（Router）

**位置**: `litellm/router.py` + `litellm/router_strategy/`

提供企業級的請求路由能力：
- **負載均衡**: 在多個 deployment 之間分散流量
- **故障轉移**: 當一個供應商失敗時自動切換
- **冷卻機制**: 暫時移除不健康的 deployment
- **策略演算法**: lowest-latency, lowest-tpm, simple-shuffle 等

### 4. 可觀測子系統（Observability）

**位置**: `litellm/integrations/` + `litellm/litellm_core_utils/litellm_logging.py`

通過 Callback 機制支援 30+ 種可觀測性工具：
- Langfuse、Prometheus、Datadog
- OpenTelemetry、Arize、Langsmith
- 自定義 Logger

### 5. Gateway 子系統（Proxy）

**位置**: `litellm/proxy/`

在 SDK 之上提供完整的 API Gateway 功能：
- API Key 管理
- 團隊/使用者預算
- 速率限制
- UI 管理介面

## 資料流向：同步 vs 非同步

LiteLLM 同時支援同步和非同步呼叫：

```python
# 同步
response = litellm.completion(model="gpt-4", messages=[...])

# 非同步
response = await litellm.acompletion(model="gpt-4", messages=[...])
```

在內部實作中：
- `completion()` → 同步 HTTP 呼叫（使用 `httpx.Client`）
- `acompletion()` → 非同步 HTTP 呼叫（使用 `httpx.AsyncClient`）

兩者共享同一套轉換邏輯（`transform_request` / `transform_response`），只有 I/O 層不同。

## 模組依賴方向

```
高層模組（依賴低層）
──────────────────
proxy/          → router.py → main.py → llms/ → custom_httpx/
                                                      │
低層模組（被依賴）                                      ▼
──────────────────                               HTTP (httpx)
                                                      │
                                                      ▼
                                               LLM Provider APIs
```

**關鍵設計原則**：依賴只能向下流動。`llms/openai/` 不會直接引用 `proxy/`，`router.py` 不會引用 `proxy/`。

## 重要的全域狀態

`litellm/__init__.py` 中定義了大量全域變數，這是本專案的特色（也是爭議點）：

```python
# litellm/__init__.py 中的關鍵全域狀態
success_callback = []       # 成功時觸發的 callbacks
failure_callback = []       # 失敗時觸發的 callbacks
input_callback = []         # 輸入時觸發的 callbacks

api_key = None             # 預設 API key
openai_key = None          # OpenAI API key
anthropic_key = None       # Anthropic API key

drop_params = False        # 是否丟棄不支援的參數
modify_params = False      # 是否自動修改參數
```

這種設計讓使用者可以透過簡單的賦值來全域設定行為：

```python
import litellm
litellm.success_callback = ["langfuse"]  # 全域啟用 Langfuse 日誌
litellm.drop_params = True               # 自動丟棄供應商不支援的參數
```

> **給初學者的提醒**：全域狀態在大型專案中通常被認為是「壞味道」（code smell），因為它讓程式的行為變得難以預測——任何地方修改了全域變數，都會影響所有地方。但 LiteLLM 選擇了這種設計是為了**使用者的便利性**。在工程中，沒有絕對的好壞，只有取捨（trade-off）。

## 自我檢測

讀完本章後，你應該能回答：

1. 一個 `litellm.completion()` 呼叫會經過哪幾個核心元件？
2. SDK 模式和 Gateway 模式的差別是什麼？什麼時候該用哪個？
3. 為什麼「依賴只能向下流動」是重要的設計原則？
4. `litellm/__init__.py` 的全域狀態有什麼優缺點？

<details>
<summary>參考答案（點擊展開）</summary>

1. `completion()` → `BaseLLMHTTPHandler` → `ProviderConfig.transform_request()` → HTTP 呼叫 → `ProviderConfig.transform_response()` → 返回 `ModelResponse`
2. SDK 模式直接在你的程式碼中呼叫；Gateway 模式透過 HTTP 中間伺服器。個人開發用 SDK，團隊 / 生產環境用 Gateway（集中管控 API Key、預算、日誌）。
3. 因為它讓每一層可以獨立修改和測試。如果底層反過來依賴上層，改一個小東西就會牽一髮動全身。
4. 優點：使用者一行程式碼就能改變行為，零配置。缺點：在多租戶 / 多執行緒場景中可能有競態問題，且難以追蹤「誰改了什麼」。

</details>
