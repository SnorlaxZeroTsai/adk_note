# 第九章：快取系統

> **本章目標**：理解 LiteLLM 的多層快取設計，以及快取在 AI 應用中扮演的不同角色。
>
> **預計閱讀時間**：15 分鐘
>
> **前置知識**：如果你用過瀏覽器快取或 CDN 的概念，本章會很直覺。核心思想：「之前算過的結果存起來，下次直接用」。
>
> **注意區分**：LiteLLM 有兩種「快取」概念——(1) **回應快取**：把 LLM 的回覆存起來避免重複呼叫；(2) **供應商的 Prompt Caching**：讓供應商對重複的 input tokens 給折扣（見第 8 章）。本章講的是第一種。

## 為什麼需要快取？

LLM API 呼叫既昂貴又慢。如果同一個問題被多次詢問，為什麼要每次都付費和等待？快取可以：
- **降低成本**：避免重複的 API 呼叫
- **降低延遲**：快取命中時接近即時回應
- **降低供應商負擔**：減少對外請求量


### DualCache 策略

```python
# litellm/caching/caching.py

class DualCache:
    """
    二級快取：先查 InMemory，未命中再查 Redis。
    寫入時同時寫入兩層。
    """
    def __init__(self, in_memory_cache=None, redis_cache=None):
        self.in_memory_cache = in_memory_cache or InMemoryCache()
        self.redis_cache = redis_cache  # 可選，未配置時只用 InMemory
    
    def get(self, key: str):
        # L1: 查 InMemory
        result = self.in_memory_cache.get(key)
        if result is not None:
            return result
        
        # L2: 查 Redis（如果已配置）
        if self.redis_cache:
            result = self.redis_cache.get(key)
            if result is not None:
                # 回填 L1
                self.in_memory_cache.set(key, result)
                return result
        
        return None
    
    def set(self, key: str, value, ttl=None):
        # 同時寫入兩層
        self.in_memory_cache.set(key, value, ttl=ttl)
        if self.redis_cache:
            self.redis_cache.set(key, value, ttl=ttl)
```

## LLM 回應快取

### 啟用方式

```python
import litellm
from litellm import completion
from litellm.caching import Cache

# 方式 1：InMemory 快取
litellm.cache = Cache()

# 方式 2：Redis 快取
litellm.cache = Cache(
    type="redis",
    host="localhost",
    port=6379,
    password="your-password",
)

# 方式 3：Redis Cluster
litellm.cache = Cache(
    type="redis",
    redis_url="redis://cluster-url:6379",
)

# 之後的呼叫會自動使用快取
response = completion(model="gpt-4", messages=[...])
# 第二次相同呼叫 → 直接從快取回傳
response = completion(model="gpt-4", messages=[...])  # 快取命中！
```

### 快取 Key 的生成

快取的 key 由請求參數決定：

```python
def _generate_cache_key(model, messages, optional_params):
    """
    相同的 model + messages + params = 相同的 cache key
    """
    cache_dict = {
        "model": model,
        "messages": messages,
        # 只包含會影響回應的參數
        "temperature": optional_params.get("temperature"),
        "max_tokens": optional_params.get("max_tokens"),
        "tools": optional_params.get("tools"),
        # ... 其他影響輸出的參數
    }
    
    # 序列化並 hash
    cache_str = json.dumps(cache_dict, sort_keys=True)
    return hashlib.sha256(cache_str.encode()).hexdigest()
```

**重要**：`temperature > 0` 的請求通常不應快取（因為每次回應都不同）。LiteLLM 預設會快取所有請求，使用者可以自行控制。

### 快取控制

```python
# 跳過快取（強制重新呼叫）
response = completion(
    model="gpt-4",
    messages=[...],
    cache={"no-cache": True},  # 不使用快取
)

# 設定自定義 TTL
response = completion(
    model="gpt-4",
    messages=[...],
    cache={"ttl": 3600},  # 快取 1 小時
)

# 程式化控制
litellm.cache.disable()   # 暫時停用
litellm.cache.enable()    # 重新啟用
```

## InMemoryCache 實作

```python
class InMemoryCache:
    """
    使用 Python dict + TTL 過期的本地快取。
    用 LRU 策略控制記憶體使用。
    """
    def __init__(self, max_size_in_memory=200):
        self.cache_dict = {}
        self.ttl_dict = {}       # key → 過期時間
        self.max_size = max_size_in_memory
    
    def get(self, key: str):
        # 檢查是否過期
        if key in self.ttl_dict:
            if time.time() > self.ttl_dict[key]:
                # 已過期，移除
                del self.cache_dict[key]
                del self.ttl_dict[key]
                return None
        
        return self.cache_dict.get(key)
    
    def set(self, key: str, value, ttl=None):
        # LRU 驅逐
        if len(self.cache_dict) >= self.max_size:
            self._evict_oldest()
        
        self.cache_dict[key] = value
        if ttl:
            self.ttl_dict[key] = time.time() + ttl
```

## RedisCache 實作

```python
class RedisCache:
    """
    Redis 基礎的分散式快取。
    支援 Redis Standalone 和 Redis Cluster。
    """
    def __init__(self, host, port, password=None, db=0):
        import redis
        self.redis_client = redis.Redis(
            host=host, port=port, password=password, db=db,
            decode_responses=True,
        )
    
    def get(self, key: str):
        result = self.redis_client.get(key)
        if result:
            return json.loads(result)
        return None
    
    def set(self, key: str, value, ttl=None):
        serialized = json.dumps(value)
        if ttl:
            self.redis_client.setex(key, ttl, serialized)
        else:
            self.redis_client.set(key, serialized)
```

## 快取在 Router 中的應用

Router 使用 DualCache 來追蹤路由狀態（不是 LLM 回應）：

```python
# Router 的快取用途
class Router:
    def __init__(self, ..., redis_url=None):
        self.cache = DualCache(
            in_memory_cache=InMemoryCache(),
            redis_cache=RedisCache(url=redis_url) if redis_url else None,
        )
    
    # 用途 1：追蹤 TPM/RPM 用量
    def _track_usage(self, deployment_id, tokens):
        key = f"tpm:{deployment_id}:{current_minute}"
        current = self.cache.get(key) or 0
        self.cache.set(key, current + tokens, ttl=60)
    
    # 用途 2：冷卻狀態
    def _set_cooldown(self, deployment_id, duration):
        key = f"cooldown:{deployment_id}"
        self.cache.set(key, True, ttl=duration)
    
    # 用途 3：延遲指標
    def _record_latency(self, deployment_id, latency_ms):
        key = f"latency:{deployment_id}"
        history = self.cache.get(key) or []
        history.append(latency_ms)
        self.cache.set(key, history[-10:], ttl=3600)  # 保留最近 10 筆
```

## Proxy 中的快取

Proxy Gateway 使用快取來加速認證：

```python
# 快取 API Key 的驗證結果
class InternalUsageCache:
    def get_key_info(self, api_key_hash):
        """
        避免每次請求都查 PostgreSQL。
        命中率高達 99%+。
        """
        return self.dual_cache.get(f"key_info:{api_key_hash}")
    
    def set_key_info(self, api_key_hash, info, ttl=300):
        """快取 5 分鐘"""
        self.dual_cache.set(f"key_info:{api_key_hash}", info, ttl=ttl)
```

## 快取模式對比

| 模式 | 速度 | 持久性 | 跨進程 | 適用場景 |
|------|------|--------|--------|---------|
| InMemory | 最快 | 無 | 否 | 開發、單進程 |
| Redis | 快 | 有 | 是 | 生產、多進程 |
| DualCache | 最快(L1命中) | 有(L2) | 是 | 最佳選擇 |

## 學習重點

1. **DualCache** 是 L1(InMemory) + L2(Redis) 的二級快取設計
2. **快取 Key** 由 model + messages + 影響輸出的參數 hash 生成
3. Router 用快取追蹤**路由狀態**（TPM、冷卻、延遲），而非 LLM 回應
4. Proxy 用快取加速 **API Key 認證**，避免每次都查資料庫
5. 快取的粒度可以透過 `cache` 參數在每次呼叫層級控制

## 什麼時候不該用回應快取？

| 場景 | 是否適合快取 | 原因 |
|------|-------------|------|
| FAQ 回答 | ✅ 非常適合 | 相同問題的答案相同 |
| 程式碼生成 | ⚠️ 看情況 | 如果 prompt 完全相同且 temperature=0，可以快取 |
| 創意寫作 | ❌ 不適合 | 使用者期望每次結果不同 |
| 聊天對話 | ❌ 不適合 | 即使內容相似，上下文不同結果也不同 |
| 資料提取/分類 | ✅ 適合 | 同樣的輸入 → 同樣的分類結果 |
| 即時資訊查詢 | ❌ 不適合 | 答案會隨時間變化 |

**經驗法則**：如果 `temperature > 0` 且使用者期望多樣性，就不應該快取。
