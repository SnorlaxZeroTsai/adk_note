# 第十一章：可觀測性與日誌

> **本章目標**：理解 LiteLLM 的 Callback 系統如何讓你追蹤 AI 應用的每一次呼叫。
>
> **預計閱讀時間**：20 分鐘
>
> **給初學者的一句話**：可觀測性就是「讓你知道系統裡正在發生什麼事」。如果你寫過 `print("呼叫了 API")`，那就是最原始的可觀測性。LiteLLM 的 Callback 系統是它的結構化、可擴展版本。

## 為什麼可觀測性重要？

在生產環境中，你需要回答：
- 哪些請求失敗了？為什麼？
- 平均延遲多少？哪些模型最慢？
- 每天花了多少錢？哪個團隊花最多？
- 使用者滿意度如何？有沒有幻覺（hallucination）？

LiteLLM 的 Callback 系統讓你無縫整合 30+ 種可觀測性工具。

## Callback 架構

```
litellm.completion() 執行
        │
        ├─── 請求前 ──→ input_callback
        │
        ├─── 請求中 ──→ (HTTP 呼叫)
        │
        ├─── 成功後 ──→ success_callback
        │              ├── Langfuse
        │              ├── Prometheus
        │              ├── Datadog
        │              └── 自定義 Logger
        │
        └─── 失敗後 ──→ failure_callback
                       ├── PagerDuty 告警
                       ├── Slack 通知
                       └── 自定義 Logger
```

## 基本設定

### 使用字串配置

```python
import litellm

# 設定成功和失敗的 callbacks
litellm.success_callback = ["langfuse", "prometheus"]
litellm.failure_callback = ["langfuse", "pagerduty"]

# 現在每次呼叫都會自動記錄
response = litellm.completion(model="gpt-4", messages=[...])
# → 自動發送到 Langfuse 和 Prometheus
```

### 在 Proxy 配置中設定

```yaml
# proxy_config.yaml
litellm_settings:
  success_callback: ["langfuse", "prometheus", "datadog"]
  failure_callback: ["langfuse", "pagerduty"]

environment_variables:
  LANGFUSE_PUBLIC_KEY: "pk-lf-..."
  LANGFUSE_SECRET_KEY: "sk-lf-..."
  LANGFUSE_HOST: "https://cloud.langfuse.com"
```

## CustomLogger — 自定義日誌

你可以建立自己的 Logger：

```python
from litellm.integrations.custom_logger import CustomLogger
from litellm.types.utils import StandardLoggingPayload

class MyCustomLogger(CustomLogger):
    def log_success_event(self, kwargs, response_obj, start_time, end_time):
        """同步成功回調"""
        print(f"✅ Model: {kwargs['model']}")
        print(f"   Latency: {end_time - start_time}")
        print(f"   Cost: ${kwargs.get('response_cost', 0):.6f}")
        print(f"   Tokens: {response_obj.usage.total_tokens}")
    
    def log_failure_event(self, kwargs, response_obj, start_time, end_time):
        """同步失敗回調"""
        print(f"❌ Model: {kwargs['model']}")
        print(f"   Error: {kwargs.get('exception', 'Unknown')}")
    
    async def async_log_success_event(self, kwargs, response_obj, start_time, end_time):
        """非同步成功回調（在 Proxy 中使用）"""
        # 可以做耗時操作（如寫入資料庫）而不阻塞回應
        await self.db.insert({
            "model": kwargs["model"],
            "latency_ms": (end_time - start_time).total_seconds() * 1000,
            "cost": kwargs.get("response_cost"),
            "tokens": response_obj.usage.total_tokens,
        })
    
    async def async_log_failure_event(self, kwargs, response_obj, start_time, end_time):
        """非同步失敗回調"""
        await self.alert_service.send_alert(
            message=f"LLM call failed: {kwargs.get('exception')}"
        )

# 註冊
my_logger = MyCustomLogger()
litellm.callbacks = [my_logger]
```

## Logging 物件 — 資料的載體

每次 LLM 呼叫都會建立一個 `Logging` 物件，它攜帶了完整的請求/回應資訊：

```python
# litellm/litellm_core_utils/litellm_logging.py

class Logging:
    def __init__(self, model, messages, ...):
        self.model = model
        self.messages = messages
        self.start_time = datetime.now()
        self.model_call_details = {}  # 儲存所有呼叫細節
    
    def pre_call(self, input, api_key, additional_args):
        """請求前：記錄輸入"""
        self.model_call_details["input"] = input
        self.model_call_details["api_base"] = additional_args.get("api_base")
        
        # 觸發 input_callbacks
        for callback in litellm.input_callback:
            callback.log_input_event(...)
    
    def success_handler(self, result, start_time, end_time):
        """成功後：觸發所有 success callbacks"""
        self.model_call_details["response"] = result
        self.model_call_details["response_cost"] = self._calculate_cost(result)
        
        for callback in litellm.success_callback:
            if isinstance(callback, CustomLogger):
                callback.log_success_event(
                    kwargs=self.model_call_details,
                    response_obj=result,
                    start_time=start_time,
                    end_time=end_time,
                )
    
    def failure_handler(self, exception, start_time, end_time):
        """失敗後：觸發所有 failure callbacks"""
        self.model_call_details["exception"] = exception
        
        for callback in litellm.failure_callback:
            if isinstance(callback, CustomLogger):
                callback.log_failure_event(
                    kwargs=self.model_call_details,
                    response_obj=None,
                    start_time=start_time,
                    end_time=end_time,
                )
```

## StandardLoggingPayload — 標準化日誌格式

LiteLLM 定義了標準化的日誌結構：

```python
class StandardLoggingPayload(TypedDict):
    # 基本資訊
    id: str                        # 請求唯一 ID
    model: str                     # 模型名稱
    custom_llm_provider: str       # 供應商
    
    # 時間資訊
    startTime: float               # 開始時間
    endTime: float                 # 結束時間
    response_time: float           # 延遲（秒）
    
    # Token 與成本
    prompt_tokens: int
    completion_tokens: int
    total_tokens: int
    response_cost: float           # 美元成本
    
    # 請求/回應內容
    messages: List[dict]           # 輸入 messages
    response: str                  # 輸出文字
    
    # 元資料
    user: Optional[str]            # 使用者 ID
    team_id: Optional[str]         # 團隊 ID
    api_key: str                   # 使用的 API Key（脫敏後）
    metadata: dict                 # 自定義元資料
    
    # 狀態
    status: str                    # "success" 或 "failure"
    error_str: Optional[str]       # 錯誤訊息（如果失敗）
```

## 主流整合工具

### Langfuse — LLM 可觀測性平台

```python
litellm.success_callback = ["langfuse"]

# 自動追蹤：
# - 每次呼叫的輸入/輸出
# - Token 用量和成本
# - 延遲
# - 模型版本
# - 可以建立 Trace（多步驟追蹤）
```

### Prometheus — 指標監控

```python
litellm.success_callback = ["prometheus"]

# 自動暴露的指標：
# litellm_requests_total          — 總請求數
# litellm_request_duration_seconds — 延遲分佈
# litellm_tokens_total             — Token 用量
# litellm_cost_total               — 成本累計
# litellm_errors_total             — 錯誤數
```

### Datadog — 企業監控

```python
litellm.success_callback = ["datadog"]

# 發送到 Datadog 的指標包含 tags：
# - model
# - provider
# - team
# - user
```

### OpenTelemetry (OTEL) — 標準化追蹤

```python
litellm.success_callback = ["otel"]

# 產生標準的 OpenTelemetry spans：
# - Span name: "litellm.completion"
# - Attributes: model, provider, tokens, cost
# - 可以整合到任何 OTEL 相容的後端
```

## Hooks（掛勾）— Proxy 的中間件

Proxy 的 Hooks 是更強大的 callbacks，可以修改請求/回應：

```python
# proxy/hooks/ 中的 hook 範例

class ContentModerationHook(CustomLogger):
    async def async_pre_call_hook(self, data, user_api_key_dict, call_type):
        """請求前：檢查輸入內容"""
        messages = data.get("messages", [])
        
        # 檢查是否包含不當內容
        for msg in messages:
            if self.is_harmful(msg["content"]):
                raise HTTPException(400, "Content policy violation")
        
        return data  # 返回（可能修改過的）請求資料
    
    async def async_post_call_success_hook(self, data, response, user_api_key_dict):
        """回應後：檢查輸出內容"""
        content = response.choices[0].message.content
        if self.is_harmful(content):
            # 替換為安全回應
            response.choices[0].message.content = "I cannot help with that."
        return response
```

## 日誌與成本追蹤的完整鏈路

```
客戶端請求
    │
    ▼
Proxy → auth → hooks(pre) → Router → SDK → Provider API
    │                                              │
    │              ← ← ← 回應 ← ← ←              │
    ▼
hooks(post) → 成本計算 → success_callback
    │                        │
    ├── Langfuse            ├── 寫入 Postgres (spend_logs)
    ├── Prometheus          ├── 更新 Redis (key_spend)
    ├── Datadog             └── 回傳 x-litellm-response-cost header
    └── Custom Logger
```

## Metadata — 自定義追蹤

你可以在每次呼叫中附加 metadata：

```python
response = litellm.completion(
    model="gpt-4",
    messages=[...],
    metadata={
        "generation_name": "user-greeting",  # Langfuse trace 名稱
        "generation_id": "gen-123",
        "trace_id": "trace-abc",             # 關聯到更大的 trace
        "tags": ["production", "v2"],         # 標籤
        "user_id": "user-456",               # 終端使用者
    },
)
```

## 學習重點

1. **三種 callback 時機**：input（前）、success（成功後）、failure（失敗後）
2. **CustomLogger** 是自定義可觀測性的標準介面
3. **StandardLoggingPayload** 確保所有整合收到一致的資料格式
4. **Hooks** 比 callbacks 更強大——可以修改請求和回應
5. **Metadata** 讓你在每次呼叫中嵌入業務上下文
6. 成本追蹤是自動的——每次呼叫都會計算並傳遞成本資訊

## 給初學者的建議：從哪裡開始？

如果你剛接觸 AI 可觀測性，這是建議的漸進路線：

```
Level 1: print/logging（開發期）
    └─ 用 litellm.set_verbose = True 看所有請求

Level 2: Langfuse（第一個生產工具）
    └─ 免費 tier 就夠用，能看到每次呼叫的 cost/latency/content
    └─ 一行配置：litellm.success_callback = ["langfuse"]

Level 3: Prometheus + Grafana（規模化監控）
    └─ 當你需要圖表、告警、SLA 追蹤時

Level 4: Custom Logger + Hooks（完全掌控）
    └─ 當現有工具不滿足需求時自己寫
```

**對初學者而言，Langfuse 是最佳起步選擇**——它專為 LLM 設計，免費 tier 支援 50K 次觀測/月，而且有漂亮的 UI 讓你看到每次對話的細節。
