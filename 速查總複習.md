## Part 1：基礎概念

### Ch.01 專案概覽與設計哲學
- LiteLLM = 100+ LLM 供應商的「萬用轉接頭」
- 以 OpenAI API 格式為統一標準（不是因為它最好，而是最通用）
- 兩層架構：SDK（轉換）+ Gateway（企業功能）
- 核心模式：適配器模式（Adapter Pattern）

### Ch.02 架構全景圖
- 請求路徑：`completion()` → `BaseLLMHTTPHandler` → `ProviderConfig.transform_request/response` → `ModelResponse`
- 五大子系統：轉換、通訊、路由、可觀測、Gateway
- 依賴只能向下流動（`proxy/` → `router` → `main` → `llms/`）
- 全域狀態設計：方便使用但有競態風險

## Part 2：核心模組

### Ch.03 核心入口 main.py
- `completion()` 五階段：驗證 → 識別供應商 → 參數轉換 → 路由分派 → 回應格式化
- 3000+ 行的巨型函式（歷史遺留，正在遷移到 `base_llm_http_handler`）
- `@client` 裝飾器負責日誌、快取、成本計算等橫切關注點
- `acompletion()` 不是獨立實作，而是把同步結果的 coroutine 拿來 await

### Ch.04 Provider 抽象層
- `BaseConfig` = 所有供應商的統一介面（策略模式）
- 核心方法：`transform_request`、`transform_response`、`get_complete_url`、`validate_environment`
- OpenAI-compatible 供應商直接繼承 `OpenAIConfig`，只覆蓋 URL/auth
- 每個供應商 = 獨立的 Config 類別，低耦合、易測試

### Ch.05 Router 路由引擎
- Model Group = 同一 `model_name` 的多個 deployment
- 路由策略：shuffle、lowest-latency、lowest-tpm、cost-based、least-busy
- Fallback 三層：同群組重試 → 跨群組 → 特殊情境（context window/content policy）
- Cooldown：失敗率超閾值 → TTL 冷卻 → 自動恢復

### Ch.06 HTTP Handler 統一通訊層
- `BaseLLMHTTPHandler`：所有供應商共用的 HTTP 處理流程
- 職責：連接池、超時控制、SSL、串流回應
- 新式供應商只需實作 `BaseConfig`，不需要碰 HTTP 層

### Ch.07 串流處理機制
- SSE 格式：`data: {JSON}\n\n`
- `CustomStreamWrapper`：統一各 Provider 不同的串流格式
- 非串流 = 等全部生成完再回傳；串流 = 邊生成邊傳

## Part 3：輔助系統

### Ch.08 成本計算引擎
- 每次呼叫結束後根據 `usage.prompt_tokens + completion_tokens` 計費
- 定價來源：內建 `model_prices_and_context_window.json` + 自定義覆寫
- 支援 prompt caching discount（重複 prefix 便宜很多）

### Ch.09 快取系統
- DualCache = L1（InMemory）+ L2（Redis）
- 讀：先 L1 → miss → L2 → miss → 原始來源；命中時回填上層
- Cache Key = hash(model + messages + 影響輸出的參數)
- Router 用快取追蹤 TPM/延遲/冷卻狀態（不是 LLM 回應）

### Ch.10 Proxy Gateway 伺服器
- FastAPI 伺服器，提供 OpenAI-compatible API endpoint
- 功能：認證、限流、預算管理、UI 管理介面
- 所有請求先過 auth → 再交給 Router → 最後到 SDK 層

### Ch.11 可觀測性與日誌
- Callback 機制：`success_callback`、`failure_callback`、`input_callback`
- 30+ 整合：Langfuse、Prometheus、Datadog、OpenTelemetry...
- `LiteLLMLoggingObj` 追蹤每個請求的完整生命週期

## Part 4：實戰與進階

### Ch.12 實戰：新增 Provider
- 四步：建目錄 → 寫 Config → 註冊到 ProviderConfigManager → 加測試
- 關鍵：`transform_request` + `transform_response` + `get_supported_openai_params`

### Ch.13 AI 工程師進階知識
- 每個設計模式都有「為什麼不用替代方案」的分析
- Adapter vs Inheritance：低耦合 + 獨立測試 vs 深繼承樹的脆弱性
- Strategy vs Chain of Responsibility：確定性 vs 動態順序
- Observer vs Middleware：解耦 vs 可預測順序
- DualCache vs only-Redis：本地快速命中 + Redis 一致性

### Ch.14 Token Budget 策略與 MCP 實作
- Budget 策略：per-key、per-team、per-model、時間窗口
- MCP Server 實作：工具定義 → schema → handler → 測試

## Part 5：快取與容錯

### Ch.15 Python LRU Cache 設計原理
- 資料結構 = Hash Map + 雙向鏈結串列，所有操作 O(1)
- CPython 用環形鏈結串列 + sentinel node
- `@lru_cache` 陷阱：不支援 mutable 參數、無 TTL、記憶體可能爆
- 在 LiteLLM：快取不變的 key string 生成，不快取會過期的狀態

### Ch.16 Cooldown Cache 設計解析
- TTL 驅動恢復：cache 過期 = 自動恢復（不需要額外恢復邏輯）
- V2 策略：基於失敗率（失敗率 > 閾值 且 請求數 > 最低門檻）
- 三級配置：全域 → Router → 個別 deployment
- DualCache：Redis 保證跨節點一致，InMemory 保證低延遲

## Part 6：網路與通訊

### Ch.17 httpx vs requests
- httpx 贏在：async 原生、HTTP/2、可替換 Transport、細粒度 timeout
- LiteLLM 做法：httpx API + aiohttp transport（好用 + 高吞吐）
- 設計啟示：介面和實作分離，可獨立替換引擎

### Ch.18 HTTP/2 與連線池設計
- HTTP/2 多路複用：一條 TCP 上多個 stream（用 stream ID 區分）
- 連線池三版演進：基礎 → +TTL/健康檢查 → +背壓/warmup/per-host
- 設計檢查清單：容量控制、複用、淘汰、健康、併發安全、背壓、生命週期

### Ch.19 Async Streaming 非同步串流
- SSE 協議 + async generator + 背壓（`Queue(maxsize=N)`）
- Proxy 模式：上游 `aiter_lines()` → parse → `StreamingResponse` → 下游
- 進階模式：fan-out（一對多）、merge（多對一）、timeout/retry
- LiteLLM `CustomStreamWrapper`：統一各 Provider 串流格式差異

## Part 7：預算與治理

### Ch.20 User Budget 查詢與管理
- 查詢：`GET /user/info` → `max_budget - spend = 剩餘`
- 階層：Key → User → Team Member → Team → Org（最嚴格的生效）
- `budget_duration` 到期自動 reset、`model_max_budget` 防單模型吃光

### Ch.21 Guardrails 安全護欄系統
- 三階段 Hook：pre_call（攔截/改寫）→ during_call（非同步審查）→ post_call（輸出過濾）
- 四種實作：Lakera（injection）、Presidio（PII）、Tool Permission、Bedrock
- 敏感資料路由：不只阻擋，還能改路到本地模型
- Guardrail 故障 ≠ 攔截（要分開追蹤）

---

## Part 8：路由與安全

### Ch.22 進階路由策略
- 9 種策略，延遲從 <0.1ms 到 ~5ms
- Complexity Router：7 維度加權評分 → 4 層模型分流，純 regex 無外部依賴
- Adaptive Router：Beta 分佈 + Thompson Sampling，自動探索/利用平衡
- Owner Cache：多輪對話只更新首次服務者的 Bandit
- 策略可以組合：Tag → Complexity → TPM/RPM → Cooldown

### Ch.23 認證與授權系統完全指南
- 認證方式：API Key（hash 存儲）→ JWT（公鑰離線驗證）→ OAuth2（即時吊銷）
- Virtual Key：使用者拿虛擬 key，真正的 Provider key 只在 Proxy 內
- RBAC 6 個角色：PROXY_ADMIN → ORG_ADMIN → TEAM → INTERNAL_USER → VIEW_ONLY
- 多租戶隔離：Org → Team → User → Key，預算每層獨立
- 速率限制：RPM + TPM 雙重、Redis 多實例同步
- auth < 1ms：分層快取（InMemory → Redis → PostgreSQL）

## 跨章核心設計原則

| 原則 | 體現 |
|------|------|
| 適配器模式 | BaseConfig 管理 100+ 供應商差異（Ch.04） |
| 雙層快取 | DualCache 貫穿所有子系統（Ch.09, 16, 22, 23） |
| TTL 驅動恢復 | Cooldown、rate limit、auth cache 都用 TTL 自動失效（Ch.16, 23） |
| 分層防禦 | 安全 = IP + Auth + RBAC + Budget + Guardrail 多層疊加（Ch.21, 23） |
| 策略可插拔 | 路由策略、guardrail、callback 都是可替換的策略物件（Ch.05, 21, 22） |
| 漸進遷移 | 新舊架構並存，新的用 base_llm_http_handler，舊的逐步搬（Ch.03） |
