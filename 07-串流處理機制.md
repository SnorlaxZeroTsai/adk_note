# 第七章：串流處理機制

> **本章目標**：理解 LLM 串流（Streaming）的原理和 LiteLLM 如何統一不同供應商的串流格式。
>
> **預計閱讀時間**：20 分鐘
>
> **前置知識**：Python 的迭代器（`for x in y`）和生成器（`yield`）。如果你用過 `for line in file`，概念是一樣的。
>
> **日常體驗連結**：你在 ChatGPT 或 Claude 網頁版看到文字「逐字出現」而不是等半天後一次跳出來——那就是串流。

## 為什麼需要串流？

LLM 生成文字是逐步進行的。如果等到全部生成完畢才回傳，使用者可能需要等待 10-30 秒。串流（Streaming）讓使用者看到文字「逐字出現」，大幅改善體驗。

```python
# 非串流：等待完整回應
response = litellm.completion(model="gpt-4", messages=[...])
print(response.choices[0].message.content)  # 等 10 秒後一次出現

# 串流：逐步接收
response = litellm.completion(model="gpt-4", messages=[...], stream=True)
for chunk in response:
    print(chunk.choices[0].delta.content, end="")  # 逐字出現
```

## Server-Sent Events (SSE) 協議

大多數 LLM API 使用 SSE 格式傳輸串流資料：

```
HTTP/1.1 200 OK
Content-Type: text/event-stream

data: {"id":"chatcmpl-1","choices":[{"delta":{"content":"Hello"}}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{"content":" world"}}]}

data: {"id":"chatcmpl-1","choices":[{"delta":{},"finish_reason":"stop"}]}

data: [DONE]
```

每一行 `data:` 就是一個 chunk，包含生成的一小段文字。

## CustomStreamWrapper — 串流統一層

`CustomStreamWrapper` 是 LiteLLM 串流處理的核心。它將各供應商不同格式的串流回應統一為 OpenAI 格式。

### 核心職責

```
供應商的原始串流
(各種格式)
    │
    ▼
┌─────────────────────────────────────┐
│        CustomStreamWrapper           │
│                                     │
│ 1. 解析供應商特定的串流格式          │
│ 2. 轉換為 OpenAI chunk 格式         │
│ 3. 追蹤完整回應（用於日誌）         │
│ 4. 計算 token 用量                  │
│ 5. 處理特殊 token 和結束信號        │
│ 6. 支援 sync 和 async 迭代         │
└──────────────────┬──────────────────┘
                   │
                   ▼
統一的 ModelResponseStream chunks
(OpenAI 格式)
```

### 初始化

```python
class CustomStreamWrapper:
    def __init__(
        self,
        completion_stream,         # 底層串流（httpx response iter）
        model: str,                # 模型名稱
        logging_obj,               # 日誌物件
        custom_llm_provider: str,  # 供應商名稱
        stream_options=None,       # 串流選項
    ):
        self.model = model
        self.completion_stream = completion_stream
        self.custom_llm_provider = custom_llm_provider
        
        # 狀態追蹤
        self.sent_first_chunk = False      # 是否已發送第一個 chunk
        self.sent_last_chunk = False       # 是否已發送最後一個 chunk
        self.complete_response = ""        # 累積的完整回應文字
        self.received_finish_reason = None # 結束原因
        
        # Token 計算
        self.chunks = []                   # 所有 chunks 的記錄
        self.send_stream_usage = True if stream_options else False
```

### 迭代協議

`CustomStreamWrapper` 同時實作了同步和非同步迭代器：

```python
class CustomStreamWrapper:
    # 同步迭代器
    def __iter__(self):
        return self
    
    def __next__(self) -> ModelResponseStream:
        """每次呼叫回傳一個 chunk"""
        try:
            chunk = self._get_next_chunk()
            if chunk is None:
                raise StopIteration
            
            # 轉換為統一格式
            processed = self._process_chunk(chunk)
            
            # 追蹤完整回應
            if processed.choices[0].delta.content:
                self.complete_response += processed.choices[0].delta.content
            
            return processed
            
        except StopIteration:
            # 串流結束，觸發 success callback
            self._handle_stream_end()
            raise
    
    # 非同步迭代器
    def __aiter__(self):
        return self
    
    async def __anext__(self) -> ModelResponseStream:
        """非同步版本，邏輯相同"""
        try:
            chunk = await self._async_get_next_chunk()
            if chunk is None:
                raise StopAsyncIteration
            return self._process_chunk(chunk)
        except StopAsyncIteration:
            await self._async_handle_stream_end()
            raise
```

## ModelResponseStream — Chunk 格式

每個串流 chunk 的格式：

```python
# 一個典型的串流 chunk
ModelResponseStream(
    id="chatcmpl-abc123",
    choices=[
        StreamingChoices(
            index=0,
            delta=Delta(
                content="Hello",       # 本次生成的文字片段
                role=None,            # 只在第一個 chunk 中設定
                tool_calls=None,      # 工具呼叫（逐步組裝）
            ),
            finish_reason=None,       # 只在最後一個 chunk 中設定
        )
    ],
    created=1700000000,
    model="gpt-4",
    usage=None,  # 只在 stream_options.include_usage 時出現
)
```

### 串流的生命週期

```
Chunk 1: delta={"role": "assistant", "content": ""}
Chunk 2: delta={"content": "Hello"}
Chunk 3: delta={"content": " "}
Chunk 4: delta={"content": "world"}
Chunk 5: delta={"content": "!"}
Chunk 6: delta={}, finish_reason="stop"
         usage={"prompt_tokens": 10, "completion_tokens": 5}
```

## 不同供應商的串流格式差異

### OpenAI
```
data: {"id":"chatcmpl-1","choices":[{"delta":{"content":"Hi"}}]}
```

### Anthropic
```
event: content_block_delta
data: {"type":"content_block_delta","delta":{"type":"text_delta","text":"Hi"}}
```

### Google Gemini
```
data: {"candidates":[{"content":{"parts":[{"text":"Hi"}]}}]}
```

`CustomStreamWrapper` 內部透過 `BaseModelResponseIterator` 的子類別來處理每種格式：

```python
# litellm/llms/base_llm/base_model_iterator.py

class BaseModelResponseIterator:
    """每個供應商的串流解析器繼承此類"""
    
    def chunk_parser(self, chunk: dict) -> GenericStreamingChunk:
        """將供應商特定的 chunk 格式轉為通用格式"""
        raise NotImplementedError


# 通用串流 chunk 格式
class GenericStreamingChunk(TypedDict):
    text: str                          # 文字內容
    tool_use: Optional[dict]           # 工具呼叫
    is_finished: bool                  # 是否結束
    finish_reason: Optional[str]       # 結束原因
    usage: Optional[dict]              # Token 用量
```

## 串流中的特殊處理

### 1. Tool Calls 的逐步組裝

工具呼叫（function calling）在串流中是逐步傳輸的：

```
Chunk 1: tool_calls=[{index:0, function:{name:"get_weather", arguments:""}}]
Chunk 2: tool_calls=[{index:0, function:{arguments:'{"lo'}}]
Chunk 3: tool_calls=[{index:0, function:{arguments:'cation'}}]
Chunk 4: tool_calls=[{index:0, function:{arguments:'": "Tokyo"}'}}]
Chunk 5: finish_reason="tool_calls"
```

`CustomStreamWrapper` 會追蹤並組裝這些片段。

### 2. Thinking/Reasoning Content

某些模型（如 Claude）會在回應中包含「思考過程」：

```python
# 追蹤 thinking blocks
self.sent_first_thinking_block = False
self.sent_last_thinking_block = False
self.thinking_content = ""

# 可選擇將 thinking 合併到 choices 中
self.merge_reasoning_content_in_choices = litellm_params.merge_reasoning_content_in_choices
```

### 3. 串流超時保護

```python
def _check_max_streaming_duration(self):
    """防止串流無限掛起"""
    elapsed = time.time() - self._stream_created_time
    if elapsed > LITELLM_MAX_STREAMING_DURATION_SECONDS:
        raise litellm.Timeout(
            message=f"Stream exceeded max duration of {max_duration}s"
        )
```

### 4. 重複 Chunk 偵測

```python
# 某些供應商可能重複發送相同的 chunk
self._repeated_messages_count = 1
# 如果連續收到相同內容超過閾值，停止串流
if repeated_count > REPEATED_STREAMING_CHUNK_LIMIT:
    raise litellm.InternalServerError("Repeated streaming chunks detected")
```

## 串流日誌的處理

串流結束時，需要彙總完整回應來觸發日誌回調：

```python
def _handle_stream_end(self):
    """串流結束的收尾工作"""
    # 1. 組裝完整的 ModelResponse（從所有 chunks）
    complete_response = self._build_complete_response()
    
    # 2. 計算 token 用量
    usage = self._calculate_usage()
    
    # 3. 觸發 success callback
    self.logging_obj.success_handler(
        result=complete_response,
        start_time=self.start_time,
        end_time=time.time(),
    )
    
    # 4. 如果 stream_options.include_usage，發送最後一個包含 usage 的 chunk
    if self.send_stream_usage and not self.sent_stream_usage:
        self._send_usage_chunk(usage)
```

## 使用模式

### 同步串流

```python
response = litellm.completion(
    model="gpt-4",
    messages=[{"role": "user", "content": "寫一首詩"}],
    stream=True,
)

for chunk in response:
    content = chunk.choices[0].delta.content
    if content:
        print(content, end="", flush=True)
```

### 非同步串流

```python
response = await litellm.acompletion(
    model="gpt-4",
    messages=[{"role": "user", "content": "寫一首詩"}],
    stream=True,
)

async for chunk in response:
    content = chunk.choices[0].delta.content
    if content:
        print(content, end="", flush=True)
```

## 學習重點

1. **SSE** 是 LLM 串流的標準協議
2. **CustomStreamWrapper** 是統一串流格式的核心組件
3. 它同時支援 `__iter__` 和 `__aiter__`，實現同步/非同步通用
4. **GenericStreamingChunk** 是內部的通用中間格式
5. 串流結束時需要彙總所有 chunks 來計算 token 用量和觸發回調
6. 特殊場景（tool calls、thinking content、超時）都需要額外處理

## 串流 vs 非串流：什麼時候該用哪個？

| 場景 | 建議 | 原因 |
|------|------|------|
| 聊天界面（面向使用者） | 串流 | 使用者體驗好，感覺更快 |
| 批次處理（後台任務） | 非串流 | 簡單、不需要即時展示 |
| 函式呼叫 / Tool Use | 非串流 | 需要完整 JSON 才能解析 |
| 長文生成（寫文章） | 串流 | 避免超時、即時看到進度 |
| 嵌入 / Embedding | 非串流 | 結果是一個向量，無法逐步 |

## 常見問題排除

<details>
<summary>Q：串流時收到空白的 chunk.choices[0].delta.content 怎麼辦？</summary>

這是正常的！第一個 chunk 通常只包含 `role: "assistant"` 而沒有 content。最後一個 chunk 也通常是空的（只有 `finish_reason`）。所以一定要加 `if content:` 的判斷。

</details>

<details>
<summary>Q：串流和非串流的成本一樣嗎？</summary>

是的。成本只跟 token 數量有關，跟傳輸方式無關。不過串流時，LiteLLM 需要在串流結束後才能計算精確的 token 用量（因為 output tokens 是逐步累積的）。

</details>

<details>
<summary>Q：如果串流過程中網路斷了怎麼辦？</summary>

你會收到一個異常。LiteLLM 的 `CustomStreamWrapper` 有超時保護（預設 30 分鐘），但不會自動重連——因為 LLM 的串流是有狀態的，中斷後無法從中間恢復。你需要重新發送整個請求。

</details>
